# 9. "얘 어때?" 투표

> 프로필 하나를 친구 여러 명에게 동시에 보내고 반응 모아보기
> 출처: Ship crew, BeReal RealMoji

## 9-1. 유저 플로우

```
[1] 피드 카드 탐색
    └→ "얘 어때?" 버튼 탭

[2] 친구 선택 (다중 선택)
    ┌─────────────────────────┐
    │ 🗳 얘 어때? 투표 보내기  │
    │─────────────────────────│
    │ ☑ 민준                  │
    │ ☑ 현우                  │
    │ ☑ 도윤                  │
    │ ☐ 서준                  │
    │─────────────────────────│
    │ 선택: 3명  (최대 10명)   │
    │ [투표 보내기]            │
    └─────────────────────────┘

[3] 친구들에게 푸시 알림
    "태양이가 네 의견을 물어봤어! 이 사람 어때?"

[4] 친구가 반응 (이모지 리액션 + 선택 코멘트)
    ┌─────────────────────────┐
    │ 지수, 26  [프로필 미리보기]│
    │─────────────────────────│
    │  😍  🤔  😐  🙅         │
    │─────────────────────────│
    │ 한 줄 코멘트 (선택)      │
    │ [   너 스타일이야!   ]   │
    │ [반응 보내기]            │
    └─────────────────────────┘

[5] 주선자 — 반응 집계 확인
    ┌─────────────────────────┐
    │ 🗳 지수 프로필 투표 결과  │
    │─────────────────────────│
    │ 😍 2명  🤔 1명           │
    │─────────────────────────│
    │ 민준: 😍 "걔 완전 너야"  │
    │ 현우: 😍                 │
    │ 도윤: 🤔 "사진 더 봐야"  │
    │─────────────────────────│
    │ [이 사람 소개하기 →]     │
    └─────────────────────────┘
```

## 9-2. DB 스키마

```sql
-- 투표 세션
CREATE TABLE vote_sessions (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    creator_id      UUID NOT NULL REFERENCES users(id),
    target_profile_id UUID NOT NULL REFERENCES users(id),
    status          VARCHAR(20) DEFAULT 'active',  -- active, closed, expired
    created_at      TIMESTAMPTZ DEFAULT now(),
    expires_at      TIMESTAMPTZ DEFAULT now() + INTERVAL '24 hours',
    closed_at       TIMESTAMPTZ
);

CREATE INDEX idx_vote_sessions_creator ON vote_sessions(creator_id, created_at DESC);

-- 투표 참여자 (친구)
CREATE TABLE vote_participants (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    session_id      UUID NOT NULL REFERENCES vote_sessions(id) ON DELETE CASCADE,
    participant_id  UUID NOT NULL REFERENCES users(id),
    reaction        VARCHAR(10),        -- emoji: heart_eyes, thinking, neutral, no
    comment         VARCHAR(100),
    responded_at    TIMESTAMPTZ,
    notified_at     TIMESTAMPTZ DEFAULT now(),
    UNIQUE(session_id, participant_id)
);

CREATE INDEX idx_vote_participants_session ON vote_participants(session_id);
CREATE INDEX idx_vote_participants_user ON vote_participants(participant_id, responded_at);
```

## 9-3. API 엔드포인트

| Method | Path | 설명 | Request Body | Response |
|--------|------|------|-------------|----------|
| `POST` | `/api/v1/votes` | 투표 세션 생성 | `{ target_profile_id, participant_ids: [] }` | `{ session_id, participants_count }` |
| `GET` | `/api/v1/votes` | 내가 만든 투표 목록 | query: `?status=active` | `{ sessions: [{ id, target, stats, created_at }] }` |
| `GET` | `/api/v1/votes/{session_id}` | 투표 상세 + 집계 | - | `{ session, target_profile, reactions: { emoji: count }, participants: [] }` |
| `POST` | `/api/v1/votes/{session_id}/respond` | 친구가 반응 보내기 | `{ reaction, comment? }` | `{ success }` |
| `PATCH` | `/api/v1/votes/{session_id}/close` | 투표 수동 마감 | - | `{ closed_at }` |
| `GET` | `/api/v1/votes/pending` | 내가 응답할 투표 | - | `{ pending_votes: [] }` |

## 9-4. 엣지 케이스

| 상황 | 처리 |
|------|------|
| 대상 프로필이 "소개받고 싶어요" OFF | 투표 생성 자체 차단. "이 사람은 지금 소개를 받지 않고 있어요" |
| 참여자 중 대상과 이미 매칭된 사람 | 해당 참여자 자동 제외 + 안내 메시지 |
| 24시간 내 응답 없음 | 해당 참여자 회색 처리. 세션은 유지 (1명이라도 응답하면 결과 열람 가능) |
| 동일 대상에 중복 투표 세션 | 최근 7일 이내 같은 대상 투표 차단. "이미 물어봤어요!" |
| 참여자가 대상의 차단 유저 | 차단 유저에게는 투표 미전달. 주선자에게 "일부 친구에게 보내지 못했어요" |
| 참여자 10명 초과 선택 | 프론트에서 선택 제한 + 서버 validation |

## 9-5. 썬구리 보상 연동

| 시점 | 대상 | 보상 | 비고 |
|------|------|------|------|
| 투표 세션 생성 | 주선자 | +5 | 기본 소개 전송과 동일 |
| 투표에 반응 | 참여자(친구) | +3 | 의견 제공 보상 |
| 코멘트까지 작성 | 참여자(친구) | +2 (추가) | 구체적 의견 보너스 |
| 투표 결과 보고 소개 전환 | 주선자 | +10 | 소개 전환 보너스 |
| 3명 이상 응답 달성 | 주선자 | +5 (보너스) | 활발한 투표 장려 |

## 9-6. 스케일링 고려사항

| 항목 | 고려 |
|------|------|
| 푸시 알림 폭주 | 참여자 10명 제한. 한 유저가 하루 받을 수 있는 투표 알림 최대 5건 |
| 투표 세션 누적 | `expires_at` 기반 자동 마감. 30일 지난 세션은 soft delete |
| 반응 집계 | 참여자 수 소규모(최대 10)이므로 실시간 집계 불필요. 단순 COUNT 쿼리로 충분 |
| 알림 배치 | 동시 발송 10건 → 알림 서비스 큐 사용 (SQS/Bull) |
| 프로필 노출 동의 | 대상자의 "소개받고 싶어요" 토글(#16) 상태 확인 필수 |

## 10-2. DB 스키마

```sql
-- 주선자 일일 소개 쿼터
CREATE TABLE daily_intro_quota (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID NOT NULL REFERENCES users(id),
    quota_date      DATE NOT NULL DEFAULT CURRENT_DATE,
    used_count      INT DEFAULT 0,
    max_count       INT DEFAULT 3,
    UNIQUE(user_id, quota_date)
);

CREATE INDEX idx_daily_quota_user_date ON daily_intro_quota(user_id, quota_date DESC);

-- 소개 대기열 (저녁 공개 전 버퍼)
CREATE TABLE intro_queue (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    intro_id        UUID NOT NULL REFERENCES introductions(id),
    recipient_id    UUID NOT NULL REFERENCES users(id),
    queued_at       TIMESTAMPTZ DEFAULT now(),
    reveal_at       TIMESTAMPTZ NOT NULL,        -- 해당 날 20:00 KST
    revealed        BOOLEAN DEFAULT false,
    expires_at      TIMESTAMPTZ,                 -- reveal_at + 24h
    viewed_at       TIMESTAMPTZ
);

CREATE INDEX idx_intro_queue_reveal ON intro_queue(reveal_at, revealed);
CREATE INDEX idx_intro_queue_recipient ON intro_queue(recipient_id, reveal_at DESC);

-- 글로벌 설정 (운영 조정용)
CREATE TABLE feature_config (
    key             VARCHAR(50) PRIMARY KEY,
    value           JSONB NOT NULL,
    updated_at      TIMESTAMPTZ DEFAULT now()
);
```

## 10-3. API 엔드포인트

| Method | Path | 설명 | Request/Response |
|--------|------|------|-----------------|
| `GET` | `/api/v1/quota/today` | 오늘 남은 소개 수 | Response: `{ used: 1, max: 3, resets_at: "2026-03-27T06:00+09:00" }` |

## 10-4. 엣지 케이스

| 상황 | 처리 |
|------|------|
| 자정~오전 6시 사이 소개 전송 | 전날 쿼터 사용. 매일 오전 6시(KST) 리셋 |
| 소개 전송 후 취소 | 공개 전(20시 전)이면 취소 가능 + 쿼터 복구. 공개 후 취소 불가 |
| 유저가 타임존이 다른 곳에 있음 | KST 기준 고정. 향후 로컬 타임존 지원 검토 |
| 공개 시점에 네트워크 오프라인 | 다음 앱 실행 시 "어제 도착한 소개가 있어요" 표시. 만료 전이면 열람 가능 |
| 하루에 소개 0건인 날 | 20시 알림 미발송. "오늘은 소개가 없었어요. 친구에게 소개를 부탁해보세요!" |
| 쿼터 부족한데 #9 투표도 하고 싶음 | 투표는 별도 행동 — 쿼터 차감 안 함. "소개 전환" 시에만 차감 |
| 프리미엄 유저 | `max_count = 5`로 상향. feature_config에서 관리 |

## 10-6. 스케일링 고려사항

| 항목 | 고려 |
|------|------|
| 20시 일괄 공개 트래픽 스파이크 | 전체 유저에게 동시 알림 → **시간 분산** 필요. 20:00~20:05 랜덤 지연 |
| 쿼터 리셋 배치 | 06:00 KST 전체 리셋 → 배치 불필요. 조회 시 날짜 비교로 처리 (lazy reset) |
| 큐 테이블 크기 | 매일 유저 수 x 평균 수신 건수. 30일 지난 데이터 아카이브 |
| 유저 수 부족 시 | 경험 확장 진입 조건: DAU 1,000+ 이상에서만 활성화. 그 전에는 제한 없는 기존 방식 유지 |
| 서버 시간 동기화 | 모든 서버 NTP 동기화 필수. reveal_at은 UTC 저장, 표시만 KST |
| AB 테스트 | feature_config로 제한 건수(3/5), 공개 시간(20시/21시) 등 운영 조정 가능 |

## 14-2. DB 스키마

```sql
-- 소개 티켓
CREATE TABLE invite_tickets (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    owner_id        UUID NOT NULL REFERENCES users(id),
    month_year      VARCHAR(7) NOT NULL,           -- '2026-03' 형식
    total_tickets   INT DEFAULT 3,
    used_tickets    INT DEFAULT 0,
    UNIQUE(owner_id, month_year)
);

CREATE INDEX idx_invite_tickets_owner ON invite_tickets(owner_id, month_year DESC);

-- 초대 코드
CREATE TABLE invite_codes (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    code            VARCHAR(12) UNIQUE NOT NULL,    -- 랜덤 12자 영숫자
    inviter_id      UUID NOT NULL REFERENCES users(id),
    ticket_id       UUID NOT NULL REFERENCES invite_tickets(id),
    status          VARCHAR(20) DEFAULT 'active',   -- active, used, expired, revoked
    invitee_id      UUID REFERENCES users(id),      -- 사용 시 기록
    created_at      TIMESTAMPTZ DEFAULT now(),
    used_at         TIMESTAMPTZ,
    expires_at      TIMESTAMPTZ DEFAULT now() + INTERVAL '30 days'
);

CREATE UNIQUE INDEX idx_invite_codes_code ON invite_codes(code);
CREATE INDEX idx_invite_codes_inviter ON invite_codes(inviter_id, created_at DESC);

-- 대기 리스트
CREATE TABLE waitlist (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    phone_number    VARCHAR(20),
    email           VARCHAR(100),
    referrer_code   VARCHAR(12),                    -- 혹시 초대 코드가 만료된 경우
    position        SERIAL,
    registered_at   TIMESTAMPTZ DEFAULT now(),
    promoted_at     TIMESTAMPTZ                     -- 대기 → 가입 허용 시점
);

CREATE INDEX idx_waitlist_position ON waitlist(position);

-- 초대 관계 (가입 후 영구 기록)
CREATE TABLE invite_relationships (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    inviter_id      UUID NOT NULL REFERENCES users(id),
    invitee_id      UUID NOT NULL REFERENCES users(id),
    invite_code_id  UUID NOT NULL REFERENCES invite_codes(id),
    created_at      TIMESTAMPTZ DEFAULT now(),
    UNIQUE(invitee_id)  -- 하나의 유저는 하나의 초대자만
);

CREATE INDEX idx_invite_rel_inviter ON invite_relationships(inviter_id);
```

## 14-6. 스케일링 고려사항

| 항목 | 고려 |
|------|------|
| 유저 풀 부족 시 | **킬 스위치 필수**. feature_config에서 `invite_required: false`로 즉시 개방 가능 |
| 대기 리스트 관리 | 대기 리스트 크기가 MAU의 30% 이상이면 월 티켓 수 상향 (3→5) 자동 조정 |
| 초대 코드 brute force | 12자 영숫자 = 62^12 조합. Rate limiting: IP당 분당 5회 |
| 티켓 월초 충전 | 매월 1일 00:00 KST 배치. lazy reset 방식 (조회 시 month_year 비교) |
| 초대 트리 깊이 | 무한 깊이 허용하되, 보상은 직접 초대(1depth)에만 지급. MLM 방지 |
| 성장률 모니터링 | `K-factor = (초대 발송 수 x 전환율)`. K < 1이면 개방 모드 전환 검토 |
| Raya 모델 vs Clubhouse 모델 | Raya: 심사위원회 (구현 비용 높음). 내친소: 친구 보증 = 자연 심사. 별도 심사 불필요 |
