# 캐주얼 추천 메시지 UX 리서치
# "친구한테 문자 보내는 느낌"을 만드는 UX 패턴

> 작성일: 2026-03-25
> 목적: 내친소 친구 소개 기능에서 "추천사"가 공식 추천서가 아닌 친구끼리 나누는 자연스러운 메시지처럼 느껴지게 하는 UX 설계 근거 수집

---

## 요약: 핵심 발견 3가지

1. **마찰이 낮을수록 전달율이 높다.** 공유 행동은 입력 필드가 아니라 버튼 하나로 시작해야 한다. 미리 채워진 메시지(pre-filled)는 전혀 없는 것보다 전환율을 높이지만, 편집 가능해야 개인적으로 느껴진다.

2. **"보낸다(forward)"는 느낌과 "추천한다(recommend)"는 느낌은 심리적으로 다르다.** Forward는 정보 전달, recommend는 사회적 책임을 동반한다. 소개팅 맥락에서는 추천자의 책임감을 낮춰야 참여율이 올라간다 — "내가 보증한다"가 아니라 "혹시 어떨까 해서 보내봤어" 프레임이 필요하다.

3. **텍스트 입력 없이도 충분히 의미 있는 신호를 보낼 수 있다.** 반응(reaction), 이모지, 원탭 신호는 "보내도 될까?" 심리적 허들을 없애며, 메시지 없는 공유도 공유다.

## 1. 소셜앱의 캐주얼 공유 메커니즘

### 1-1. TikTok: "Shared Feed"와 친구에게 보내기

**어떻게 작동하나**

TikTok은 2025년 12월에 "Shared Feed" 기능을 출시했다. 두 사람이 서로 팔로우하고 초대를 수락하면, 1:1 DM 안에 매일 15개의 영상이 담긴 공동 피드가 생성된다. 피드는 두 사람의 시청 이력, 좋아요, 관심사를 기반으로 자동 생성된다. 둘 다 각 영상에 반응하거나 채팅할 수 있고, 전부 소진 후 "Shared Likes" 히스토리로 공통 취향을 확인할 수 있다.

**단일 영상 "친구에게 보내기" 플로우**

```
영상 시청 중
→ 오른쪽 하단 공유 아이콘(화살표) 탭
→ 수신자 선택 (최근 DM 목록 자동 노출)
→ [선택] 메시지 텍스트 추가 (공란이어도 발송 가능)
→ 보내기
```

핵심 UX 결정: 메시지 없이도 즉시 전송 가능. 텍스트는 선택이지 필수가 아니다.

**내친소에 적용할 포인트**
- 프로필 카드 공유 시 메시지 텍스트는 선택 (필수 아님)
- 수신자 선택 UI에서 최근 대화 목록 자동 노출 → 탭 1번으로 전달

### 1-2. Instagram DM 공유: 자연스러움의 원천

Instagram에서 게시물을 DM으로 보낼 때 텍스트를 추가하는 행동은 이미 수억 명에게 훈련된 습관이다. 공유 버튼(종이비행기 아이콘) → 수신자 선택 → 메시지 입력창이 이미 열려 있음 → 보내기. 텍스트 입력창이 선택지로 열려 있다는 점이 핵심 — "뭔가 말해도 되는 공간"이 준비되어 있지만 비워도 된다.

**실패 사례: 저장된 게시물 DM 공유**

저장 게시물을 DM에서 공유하려면 현재 앱 밖으로 나갔다 들어와야 하는 다단계가 필요하다. 이 사례는 마찰이 공유 의도를 얼마나 죽이는지를 역으로 증명한다.

**교훈**: 공유 행동은 현재 화면에서 즉시 시작되어야 한다. 다른 탭이나 메뉴로 이동하는 순간 사용자는 행동을 포기한다.

### 1-3. Snapchat: 에피소드성이 만드는 낮은 심리적 부담

Snapchat이 공유를 가볍게 느끼게 하는 이유는 기술적 설계가 아니라 **심리적 조건** 때문이다.

- **사라지는 메시지**: 영구 기록이 아니기 때문에 "틀린 말 하면 어쩌지"라는 부담이 없다
- **카메라 최우선 진입**: 앱을 열면 바로 카메라 → "뭔가를 공유하러 들어왔다"는 맥락이 기본값
- **제스처 기반 탐색**: 텍스트 대신 스와이프로 대부분의 행동 완료 → 인지 부하 최소화
- **즉흥성 장려**: 정제되지 않은 콘텐츠가 기본이기 때문에 "잘 만들어야 한다"는 압박이 없음

**소개팅 추천메시지에 적용**: 추천 메시지가 영구 기록으로 남는다면 추천자는 더 신중해진다. "이 메시지는 상대방도 볼 수 있어요"라는 표시가 있으면 오히려 부담이 생긴다. 메시지의 가시성과 지속성을 어떻게 설계하느냐가 참여율에 직접 영향을 준다.

### 1-4. BeReal RealMoji: 사진으로 반응하는 저마찰 신호

BeReal의 RealMoji는 "반응 사진"이다. 친구의 BeReal을 보고 이모지를 탭하면 셀카 카메라가 열리고, 즉석 표정 사진이 그 이모지 자리를 대체한다.

**UX 플로우**

```
친구의 BeReal 확인
→ 하단 이모지 탭 (또는 더블탭)
→ RealMoji 선택 (6가지: 엄지척, 웃음, 놀람, 하트눈, 웃음-눈물, 자유)
→ 셀카 촬영
→ 즉시 전송
```

규칙: BeReal 하나당 RealMoji 1개만 가능.

**왜 이것이 "캐주얼"한가**
- 텍스트 입력 없음
- "좋아요"보다 더 개인적이지만 댓글보다 훨씬 쉬움
- 내 얼굴이 들어가기 때문에 공식 반응보다 친근하게 느껴짐
- 제약(1개만)이 오히려 결정 피로를 없앰

**내친소 적용 아이디어**: 주선자가 프로필 카드를 보낸 후, 수신자(잠재 매칭 대상)가 반응으로 짧은 셀카나 이모지를 돌려보내는 구조 → 텍스트 없는 "관심 신호"

### 1-5. iMessage Tapback: 원탭으로 충분한 이유

Tapback은 메시지 버블을 길게 눌러 6개 반응 중 하나를 선택하는 기능이다. 하트, 엄지척, 엄지다운, HaHa, !!, ?. iOS 18에서는 전체 이모지 라이브러리로 확장됐다.

**설계 원칙**

1. **메시지당 1개 제한**: 제약이 결정을 빠르게 만든다
2. **특정 메시지에 고정**: 대화 흐름을 끊지 않고 반응 가능
3. **모두가 볼 수 있음**: 그룹 대화에서 사회적 신호 역할
4. **타이핑 불필요**: 상대가 보낸 직후 즉시 처리 가능

**왜 이것으로 충분한가?**

원탭 반응이 "아무 말 안 함"과 "긴 답장"의 중간 공간을 채운다. "읽었는데 대답하기엔 별로지만, 무시하기도 싫어"라는 상황에 정확히 맞는다. 소개팅 맥락에서 이 공간이 중요한 이유: 친구에게 소개받은 프로필을 보고 "글쎄..."라고 느낄 때 거절 문자 쓰기 싫은 상황 — 원탭 반응이 해결한다.

### 1-6. KakaoTalk 스티커와 빠른 답장

KakaoTalk은 2011년 스티커를 도입하여 텍스트 타이핑 없이도 감정 전달이 가능한 언어를 만들었다. 현재 30만 개 이상의 이모티콘이 있고, 이모티콘 선물이 한국 친구 간 일상적 문화다.

**왜 스티커가 캐주얼한가**

- 타이핑 없이 감정 전달
- "공식적 언어"가 아님 — 친한 사람끼리 쓰는 신호
- 이모티콘 선물 = 관계 유지 비용 최소화
- AI 스티커 추천: 대화 맥락을 분석해 적절한 이모티콘 자동 추천

**학술 근거**: ACM Human-Computer Interaction 2024 연구에 따르면, 스티커 추천 시스템에서 사용자가 스티커를 선택하는 이유는 "감정 정확도"보다 "내가 그 사람에게 어떻게 보이고 싶은가"가 더 중요하다. 즉, 스티커는 감정 표현이자 관계 표현이다.

## 2. 음악/엔터테인먼트 추천: 캐주얼 메시지로서의 공유

### 2-1. Spotify Messages: 2025년 8월 출시

Spotify는 2025년 8월 인앱 다이렉트 메시징을 출시했다. 출시 이후 4,000만 명이 3억 4,000만 건의 메시지를 발송했다.

**공유 플로우**

```
음악 재생 중 (Now Playing 화면)
→ 공유 아이콘 탭
→ 친구 선택 (이미 Blend, Jam, 협업 플레이리스트로 연결된 사람 우선 노출)
→ 보내기 (메시지 선택 사항)
```

2026년 1월 확장 기능: 친구가 지금 무엇을 듣는지 보이는 "Listening Activity" — 친구의 현재 재생곡을 탭하면 내 라이브러리에 추가하거나 같이 듣기 시작하거나 이모지로 반응 가능.

**핵심 UX 결정**: 음악 공유가 "들어봐"라는 텍스트 없이도 "내가 지금 이걸 듣고 있어"라는 존재감 공유로 작동한다. 추천 행위가 실시간 상태 공유로 치환된다.

**내친소 적용 아이디어**: "지금 이 프로필 보고 있어"라는 실시간 신호를 친구에게 보낼 수 있는 구조 — 텍스트 추천이 아닌 행동 공유.

---

### 2-2. Apple Music SharePlay: 함께 듣기

SharePlay는 FaceTime 통화 중 Apple Music을 함께 듣는 기능이다. iOS 16부터 iMessage에서도 SharePlay 링크를 보낼 수 있다.

**왜 이것이 "추천"이 아닌 "함께하기"인가**

SharePlay의 프레이밍은 "이 음악 들어봐"가 아니라 "지금 같이 들을래?"다. 비동기 추천이 아니라 동기적 공유 경험으로 설계되어 있다. 수신자 각자에게 Apple Music 구독이 필요하다는 제약이 있지만, 공유 컨트롤(재생, 순서 변경, 다음 곡)을 양쪽이 모두 쥔다는 점이 일방적 추천과 다른 감각을 만든다.

---

### 2-3. Letterboxd: 왜 친구 리뷰가 형식적이지 않은가

Letterboxd는 영화 리뷰 플랫폼이지만 리뷰 문화가 극도로 캐주얼하다. 한 줄짜리 농담 리뷰와 깊이 있는 비평이 동등하게 존재한다.

**캐주얼함의 설계 요소**

1. **별점 중심**: 긴 글 없이 0.5점 단위 별점 5개로 의견 표현 가능
2. **"봤어요" 체크만 해도 됨**: 리뷰 없이 시청 기록만 남겨도 친구 피드에 공유됨
3. **친구 활동 피드**: 내가 팔로우하는 사람이 방금 본 영화가 실시간으로 보임 — 리뷰가 없어도 "얘도 봤구나"라는 신호
4. **리뷰 길이 제한 없음**: 한 줄도 OK, 에세이도 OK → 표현 부담이 없음
5. **유머가 기본**: 플랫폼 문화 자체가 "진지한 비평"보다 "친구랑 영화 얘기하는" 느낌

**내친소에 적용**: 추천사가 "잘 작성된 소개 글"이어야 한다는 암묵적 기대를 없애야 한다. 한 단어, 이모지, 짧은 농담도 유효한 추천으로 취급되는 문화 설계가 필요하다.

## 3. 친구 추천이 행동을 이끄는 비데이팅 앱들

### 3-1. Google Maps: 장소 공유에 노트 추가

Google Maps의 리스트 기능은 장소에 개인 메모를 붙여 공유할 수 있다. 공동 편집이 가능하며, 각 장소에 누가 추가했는지 이름과 프로필 사진이 표시된다.

**핵심 UX**: 장소 추가 + 짧은 설명(예: "여기 파스타 진짜 맛있음")이 자연스러운 이유는 **리스트라는 컨테이너**가 있기 때문이다. 리스트 안에서 짧은 노트는 공식 리뷰가 아니라 큐레이션 맥락에서 나온 개인 메모로 읽힌다.

**배울 점**: 추천 메시지는 "편지"가 아니라 "라벨"처럼 느껴져야 한다.

---

### 3-2. Airbnb Wishlist 공유: 발견된 실패 패턴

Airbnb Wishlist는 공동 편집이 가능하고, 각 숙소에 노트와 투표(좋아요/별로)를 달 수 있다. 하지만 실제 사용자 중 Wishlist 공유 기능을 아는 비율은 약 9%에 불과하다.

**실패 원인 분석**

- Wishlist가 "개인 저장소"처럼 보여서 공유 가능한 것인지 모름
- 공유하라는 진입점이 기능 안에 묻혀 있음
- "친구에게 보낼래요?" 같은 진입점 넛지가 없음

**교훈**: 공유 기능이 존재한다고 사용되는 것이 아니다. 공유를 하고 싶게 만드는 **트리거 맥락**이 필요하다. 소개팅 앱에서 "이 카드 괜찮은데 OO한테 어울릴 것 같다"는 생각이 드는 순간, 그 생각이 사라지기 전에 공유 버튼이 보여야 한다.

## 4. 데이팅 인접 앱의 캐주얼 사회적 증거

### 4-1. Tinder Matchmaker: 친구가 매칭을 대신 고르는 기능

**어떻게 작동하나**

Tinder 유저가 4시간짜리 세션 링크를 친구 최대 15명에게 공유한다. 친구는 Tinder 계정 없이도 게스트로 참여할 수 있다 (나이 확인 후). 친구는 4시간 내에 프로필에 "좋아요"를 누를 수 있고, 채팅은 불가능하다. 세션 종료 후 "친구 추천" 태그가 붙은 프로필이 별도로 표시되고, 최종 결정은 본인이 한다.

**UX 분석: 잘 된 것과 아쉬운 것**

잘 된 것:
- 친구가 Tinder 계정 없이도 참여 가능 → 초대 장벽 낮음
- 최종 결정권은 본인 → 추천자 부담 줄어듦
- "친구 추천" 태그로 다른 카드와 구분됨 → 소셜 증거 시각화

아쉬운 것:
- 4시간 세션 제한이 너무 짧아 일정 조율이 어려움
- 친구가 추천 이유를 남길 수 없음 → 추천이 "좋아요" 클릭과 다를 바 없음
- 추천자가 결과를 알 수 없음 → 피드백 루프 없음 → 재참여 유인 없음

---

### 4-2. Hinge: 저마찰 긍정 신호의 설계

Hinge의 모든 좋아요 신호는 "프로필 전체"가 아니라 특정 사진이나 답변에 반응하는 구조다.

**왜 이것이 중요한가**

"당신 좋아요"가 아니라 "이 사진 웃겼어요" → 더 구체적이고 덜 무서운 신호. 수신자 입장에서 칭찬이 추상적이지 않아서 읽기 쉽고, 발신자 입장에서 뭔가 특정해서 말하는 게 "진지한 고백"보다 낮은 부담이다.

**Rose(로즈)**: 특별한 관심 신호로, 상대방의 "좋아요" 큐에서 최상단에 표시된다. 일반 좋아요와 시각적으로 구분되어 강도 차이를 표현할 수 있다.

**소개팅 추천에 적용**: "이 사람 좋을 것 같아"보다 "이 사람 사진 3번째 거 너 취향이잖아"가 더 자연스럽다. 추천 메시지를 자유 텍스트 대신 프로필의 특정 요소를 태그하는 방식으로 설계하면 더 구체적이고 덜 부담스럽다.

---

### 4-3. Coffee Meets Bagel: 큐레이션 프레이밍

CMB는 하루 정오에 소수의 "Bagel"(추천 매칭)을 제공한다. 여성에게는 6명, 남성에게는 최대 21명.

**캐주얼함의 원천**

- 알고리즘이 골랐다는 프레이밍 → "왜 이 사람?"이라는 질문이 자연스럽게 약해짐
- 정오라는 특정 시간 → 공유와 리텐션을 동시에 설계
- 소수 → 한 명 한 명이 더 진지하게 보임

**친구 추천에 적용**: "오늘 내 친구 추천" 프레이밍으로 하루에 1~3명만 소개 가능하게 제한하면 오히려 더 가치 있는 신호로 인식된다.

## 5. 캐주얼 vs 공식 추천의 심리학

### 5-1. 왜 친구 추천이 공식 리뷰보다 신뢰받는가

**수치 근거**
- 소비자의 88%는 아는 사람의 추천을 모든 형태의 마케팅보다 더 신뢰한다 (Buyapowa, 2025)
- 친구/가족 추천 vs 브랜드 메시지: 신뢰도 5배 차이 (LinkedIn Research)
- 추천 유저는 광고 유저보다 CAC 50% 낮고 LTV 16% 높다 (업계 평균)

**심리적 메커니즘 3가지**

1. **이해충돌 없음 (No vested interest)**: 친구는 나를 팔려고 추천하는 게 아니다. 공식 리뷰어는 뭔가 이득이 있을 수 있다는 의심이 생긴다.

2. **취향 유사성 (Similarity)**: 나와 비슷한 사람이 좋아하면 나도 좋아할 가능성이 높다. 친구는 나를 알기 때문에 AI 알고리즘보다 더 정확하게 "이 사람이 좋아할 것"을 알 수 있다.

3. **감정적 투자 (Perceived investment)**: 친구가 추천했다는 것은 나를 위해 시간을 썼다는 의미다. 이게 신뢰의 핵심 단서다.

---

### 5-2. "추천사"를 "공식 추천서"처럼 느끼게 만드는 요소들 — 피해야 할 것들

| 요소 | 공식적으로 느껴지는 이유 | 대안 |
|---|---|---|
| 긴 입력창 | 긴 글을 써야 한다는 압박 | 짧은 입력창 또는 캐릭터 제한 표시 |
| "추천사 작성" 레이블 | 공문서 느낌 | "한마디 남기기", "어떤 점이 떠올랐어?" |
| 필수 입력 | 쓰지 않으면 진행 불가 | 선택 사항으로 설계 |
| 제출 후 수정 불가 | 신중하게 써야 한다는 인식 | 수정 가능 |
| 상대방에게 즉시 공개 | 부담 증가 | "먼저 나한테만 보이고, 수락 시 공개" |
| 격식체 요청 | "공식적으로 추천드립니다" 톤 강제 | 구어체 예시 프롬프트 |

---

### 5-3. 음성 메시지 vs 텍스트: 친근감의 차이

**연구 결과**

음성 통화, 영상 통화, 음성 채팅이 포함된 인터랙션은 텍스트 기반 인터랙션보다 더 강한 사회적 유대를 형성한다 (심리학 연구). 음성에는 억양, 속도, 감정이 담기기 때문에 텍스트에서 전달하기 어려운 뉘앙스가 전달된다.

**그러나 음성 메시지의 실용적 문제**

- 68%의 사용자가 음성 메시지를 완전히 이해하기 위해 두 번 이상 들어야 한다고 응답 (YouGov, 2023)
- 공공장소에서 듣기 불편
- 어떤 내용인지 미리 볼 수 없어 수신 거부 심리 발생

**결론**: 음성 추천 메시지는 친근하지만, 수신자에게 더 많은 노력을 요구한다. 첫 추천 단계보다 관계가 어느 정도 형성된 후 단계에 적합하다.

---

### 5-4. 짧은 신호 vs 긴 추천의 전환율

직접 비교 데이터는 없으나, 추론 가능한 패턴:

| 신호 유형 | 발신자 부담 | 수신자 신뢰도 | 전환율 추정 |
|---|---|---|---|
| 이모지 반응만 | 매우 낮음 | 낮음 | 높음 (보내는 사람 많음) |
| 원탭 + 미리채워진 메시지 | 낮음 | 중간 | 높음 |
| 2~3줄 자유 텍스트 | 중간 | 높음 | 중간 |
| 5줄 이상 자유 텍스트 | 높음 | 매우 높음 | 낮음 (쓰는 사람 적음) |
| 음성 메시지 | 높음 | 매우 높음 | 매우 낮음 |

**최적 설계**: 낮은 진입 장벽(이모지 또는 원탭) + 선택적 텍스트 추가 옵션. 쓰고 싶은 사람은 더 쓸 수 있고, 이모지만 보내도 유효한 신호로 처리.

## 6. UX 패턴 종합: 공유를 자연스럽게 만드는 결정들

### 6-1. 미리 채워진 메시지 (Pre-filled templates)

**바이럴 리퍼럴 연구에서 밝혀진 것**

원탭 공유 + 미리 채워진 메시지는 바이럴 리퍼럴 프로그램에서 표준 권고 사항이다. 그러나 중요한 조건이 있다: **편집 가능해야 개인적으로 느껴진다.**

미리 채워진 메시지는 시작점을 낮추지만, 수정 불가능하면 "기계적 메시지"가 된다. 수정 가능한 미리채움은 "내가 쓴 것처럼" 느껴진다.

**한국어 데이팅 맥락 예시**

나쁜 예: "OOO님이 내친소에서 당신을 만나보라고 추천했습니다."
좋은 예 (편집 가능): "야 이 사람 너 취향 아니야? 한번 봐봐"

---

### 6-2. 이모지/반응 전용 옵션

Facebook Reactions 출시 이후 첫 1년 내 3,000억 건의 반응이 공유됐다 (하루 8억 건). 이모지를 포함한 게시물은 포함하지 않은 것보다 공유율 33%, 인터랙션 57% 높다.

**설계 원칙**: 반응 옵션을 공유 플로우의 첫 단계로 배치. "이 사람 어때?" + 이모지 반응 → 텍스트 없이도 의견 전달 가능.

---

### 6-3. "포워딩"이 아닌 "소개하기" 프레이밍

**프레이밍 효과 (Framing Effect)**

심리학자 Kahneman과 Tversky의 연구에 따르면, 논리적으로 동일한 선택도 표현 방식에 따라 다른 행동을 유발한다. "추천"은 사회적 책임 부담이 있지만 "소개"는 중립적이다. "보증"은 결과 책임을 암시하지만 "한번 봐봐"는 가볍다.

| 표현 | 심리적 무게 | 참여율 예상 |
|---|---|---|
| "추천서 작성하기" | 무거움 | 낮음 |
| "친구에게 추천하기" | 중간 | 중간 |
| "소개해주기" | 가벼움 | 높음 |
| "한번 보내보기" | 매우 가벼움 | 높음 |
| "이 사람 어때?" | 최소 부담 | 가장 높음 |

---

### 6-4. 공유 타이밍: 생각이 나는 순간에 잡아라

Airbnb Wishlist의 실패(9% 인지율)는 공유 진입점이 기능 안에 묻혀 있었기 때문이다. Google Maps는 리스트를 볼 때마다 공유 버튼이 보인다.

**내친소에 적용**: 주선자가 피드에서 카드를 보는 순간, "이 사람 OO한테 어울릴 것 같다"는 생각이 드는 그 순간에 공유 버튼이 보여야 한다. 카드를 다 보고 넘긴 후에 "혹시 친구에게 소개하고 싶은 사람이 있었나요?" 알림을 보내는 것은 너무 늦다.

**Tinder Matchmaker의 교훈**: 세션 링크를 만들어서 친구에게 보내는 방식은 맥락 전환(context switch)을 강제한다. "지금 보는 이 카드를, 지금 바로" 보낼 수 있는 구조가 훨씬 자연스럽다.

---

### 6-5. 양방향 수락 구조 (LinkedIn 방식)

LinkedIn의 상호 소개 기능: 소개를 보내기 전에 수신자에게 "받을래요?" 먼저 묻는다. 양쪽이 동의해야 연결된다.

**왜 이것이 중요한가**

- 소개자 보호: 동의 없이 프로필을 공유당한 사람이 불쾌감을 느끼면 내친소 신뢰 훼손
- 수신자 자율성: 친구의 추천이지만 최종 선택은 본인

**흐름 설계 제안**

```
A (주선자) → "이 사람 어때?" + 프로필 일부 → B (소개받을 친구)
B → "볼게" / "괜찮아"
B가 수락 → 프로필 전체 공개
B가 수락 + 좋아요 → 상대방(C)에게 "OO님의 친구가 관심 있어요" 알림
C → 수락/거절
양쪽 수락 → 연결
```

## 7. 내친소 추천 메시지 설계 권고안

### 핵심 원칙

**원칙 1: "추천서"가 아닌 "문자" 톤**
입력창 레이블, 플레이스홀더 텍스트, 버튼 레이블 모두 구어체. "추천 내용을 입력하세요"가 아니라 "어떤 점이 생각났어? (선택)"

**원칙 2: 없어도 괜찮다**
메시지 없이 프로필만 보내도 유효한 행동. 이모지 하나도 유효한 신호. 긴 추천사는 쓰고 싶을 때만.

**원칙 3: 카드를 보는 순간이 공유 타이밍**
피드 카드에 공유 버튼이 바로 보여야 한다. 다른 화면으로 이동하면 생각이 사라진다.

**원칙 4: 발신자와 수신자 모두의 부담을 낮춰라**
발신자: "틀린 추천을 하면 어쩌지" 부담 → "소개만 했을 뿐, 결정은 네가 해"
수신자: "이걸 어떻게 반응해야 하지" 부담 → 원탭 반응 옵션 제공

**원칙 5: 짧고 구체적인 추천이 길고 막연한 것보다 낫다**
Hinge의 접근처럼 — "이 사람 좋을 것 같아"보다 "이 사진 너 취향이잖아"

### 최소 viable 플로우 (MVP)

```
피드 카드 보는 중
→ "친구에게 소개하기" 버튼 (카드 하단, 작게)
→ 친구 선택 (최근 대화 목록, 또는 연락처)
→ [선택] 빠른 반응: 이모지 3개 (예: 😏 👀 🔥) 또는 짧은 텍스트
→ 보내기
→ 수신자에게 "OO님이 이 사람 어떤지 물어봐" 알림 (카드 일부 미리보기 포함)
→ 수신자: 볼게 / 됐어 탭
→ "볼게" 선택 시 프로필 전체 공개
```
