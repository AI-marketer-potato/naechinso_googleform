### 1-1. Ship (Match Group × Betches Media, 2019~2022)

**개요**
- Match Group과 미디어 브랜드 Betches가 공동 개발한 크루 기반 매칭 앱
- 2019년 1월 iOS 출시, Match Group 패밀리 중 가장 빠르게 성장한 앱으로 시작
- 2022년 9월 30일 서비스 종료

**핵심 메커니즘**
- **Crew 시스템**: 사용자가 친구를 "크루(crew)"에 초대 → 크루 멤버가 당사자를 위해 스와이프 대리 수행
- **Crew Chat**: 잠재적 매칭 상대를 두고 그룹 채팅으로 의견 나눔
- **결과**: 전체 매칭의 53%가 크루의 대리 스와이프에서 발생 — 핵심 기능은 기술적으로 작동함

**왜 실패했나 — PO 관점**

| 원인 | 설명 |
|---|---|
| 주선자 리텐션 제로 | 친구 매칭이 성공하면 당사자는 앱을 떠나지만, 주선자는 계속 써야 하는 이유가 없었다 |
| 역할 분리 문제 | 주선자 역할과 소개받는 역할이 분리되어 있어, 이미 연애 중인 사람이 쓸 이유가 없었다 |
| 보상 부재 | 성사시켜줘도 주선자에게 돌아오는 인센티브가 없었다 |
| 브랜드 정체성 약화 | Match Group의 다른 앱과 차별화 포인트가 시간이 지나면서 희석됨 |

**CEO 발언 (Bernard Kim, 2022)**: "We must focus our investment on brands that are growing and pull back in areas that are not driving growth."

**PO 교훈**: 친구 소셜 기능이 기술적으로 잘 작동해도, **주선자가 계속 써야 할 이유**가 없으면 앱이 유지되지 않는다. 역할 통합이 선행 조건.

### 1-2. Wingman (2018~현재 운영 중)

**개요**
- 친구가 대신 프로필을 작성하고 매칭하는 앱
- 자신의 소개도 받으면서 친구의 소개도 도와주는 이중 구조

**핵심 메커니즘 — Dual Mode**
```
Wingee (소개받는 사람) 역할:
  → 마음에 드는 카드 발견 → 오른쪽 스와이프
  → Wingman에게 nudge(연락 요청)
  → Wingman이 소개 진행

Wingman (주선자) 역할:
  → 친구 프로필 직접 작성 ("good/bad/ugly" 항목으로 솔직하게)
  → 친구를 위해 스와이프
  → 소개 성사 시 리더보드 포인트 획득
```

**핵심 UX 결정**
- 프로필을 **친구가 직접 작성** → 자기 PR보다 신뢰도 높음 (Wingman 내부 조사 기준 80% 높은 매칭률 주장)
- **복수 Wingman 허용** → 여러 친구가 동시에 주선 경쟁 → 리더보드 경쟁 생성
- 역할 전환 자유 → 오늘은 주선자, 내일은 소개받는 쪽 → 이탈 방지

**Ship과의 결정적 차이**

| 구분 | Ship | Wingman |
|---|---|---|
| 역할 구조 | 당사자 + 크루 분리 | 하나의 앱에서 역할 전환 |
| 주선자 보상 | 없음 | 리더보드 포인트 |
| 지속 생존 | 2022년 종료 | 현재까지 운영 |

**PO 교훈**: Wingman이 Ship보다 오래 살아남은 이유는 **역할 통합과 주선자 리워드**. 이 두 가지가 없으면 주선 기능은 유지되지 않는다.

### 1-3. Hinge Matchmaker (2017 — 단독 앱, 이후 통합)

**개요**
- Hinge가 2017년 9월 출시한 별도 스탠드얼론 앱
- "연애 중인 사람도 싱글 친구를 위해 데이팅앱에 참여할 수 있다"는 컨셉

**UX Flow (3단계)**
```
Step 1: 내 Facebook 친구 중 Hinge 사용자를 확인
Step 2: 두 명을 지정 → 매칭 추천
Step 3: 두 사람 모두에게 아이스브레이커 메시지 자동 전송 → 연결 성사
```

**핵심 설계 원칙**
- **Facebook 소셜 그래프 활용**: 이미 아는 사람들 사이에서 연결 → 신뢰도 확보
- **아이스브레이커 자동 발송**: 소개받는 사람이 어색함 없이 대화 시작 가능
- **초대 링크 흐름**: 비사용자에게 초대 링크 발송 → 신규 유저 유입 루프 생성

**한계와 이후 변화**
- 스탠드얼론 앱이 메인 앱과 따로 존재 → 사용자 분산 문제
- 이후 Hinge는 메인 앱 내 "프로필 공유(Send Profile)" 기능으로 흡수
- 2025년 2월에는 "Match Note" (매칭 상대에게 개인 정보 비공개 공유) 기능 출시

**PO 교훈**: 아이스브레이커 자동 발송이 핵심 마찰(어떻게 말 걸지)을 제거했다. **소개 행동의 마찰을 낮추는 것이 전환율 결정**.

### 1-4. Cerca (2025 — Gen Z 소셜 그래프 데이팅)

**개요**
- 2025년 Georgetown/USC 학생 공동 창업
- Ex-Tinder, Match, OkCupid, Grindr 임원들이 투자 (시드 $1.6M)
- 핵심 콘셉트: "date mutuals, not strangers (아는 사람의 아는 사람과 만난다)"

**핵심 메커니즘**
```
1. 앱 설치 → 연락처 자동 동기화
2. 매일 4명의 추천 제공 (각각 1촌 또는 2촌 연결 필수)
3. 프로필 표시 순서: 공통 친구 먼저 → 배경 → 사진 (외모보다 연결 강조)
4. Like 시: 상대 피드에 프로필 부스트 (누가 먼저인지는 비공개)
5. 저녁에 매칭 공개 → "누가 먼저였는지" 서로 모름 → 부담 감소
```

**핵심 설계 원칙**
- **연락처 동기화 강제 → 초대 행동 자연 발생**: 앱이 작동하려면 네트워크가 필요
- **하루 4개 제한**: Tinder식 스와이프 피로 해소 → 품질 집중
- **양방향 비공개 매칭**: 상대가 누가 먼저 좋아했는지 모름 → 안전하게 좋아할 수 있음
- **안전성**: 공통 친구를 통해 사전 검증 가능 → 여성 유저 신뢰도 상승

**Gen Z 타겟팅**
- 데이팅앱 피로도 해소 (무한 스와이프 → 하루 4명)
- "낯선 사람"이 아닌 "주변인" → 안전성 + 진정성
- 지인 네트워크 기반 → 앱 밖 대화(카카오, 인스타)로 자연 연결

**PO 교훈**: **프로필에서 사진보다 공통 친구를 먼저 보여주는 것** 자체가 신뢰 신호. 내친소에서 "OO님과 친구" 뱃지를 상단에 배치하는 이유가 여기 있다.

### 1-5. S'More (2019~현재)

**개요**
- "Love Is Blind" 앱 버전 — 외모 블러 후 대화로 점진적 언블러
- 2021년 멤버십 2배 성장, "소셜 기능"보다는 슬로우 리빌 메카닉이 핵심

**슬로우 리빌 메카닉**
```
1. 매칭 → 사진은 완전 블러 처리 상태
2. 채팅 → 대화할수록 사진 조각씩 언블러
3. 아이콘 상호 작용("wink") → 추가 블러 해제
4. 소셜 미디어 피드도 언블러 후 공개
```

**주요 수치**
- 프로필 상호작용의 35% 이상이 셀럽 데이팅 콘텐츠 시청 후 발생
- 2021년 멤버십 2배 증가

**PO 교훈**: 외모 우선 노출이 여성 이탈의 주요 원인 중 하나. **프로필 노출 순서를 조정하는 것만으로도 매칭 품질과 여성 리텐션이 달라질 수 있다**.

### 1-6. Thursday (영국, 2021~현재)

**개요**
- "목요일 하루만 활성화"되는 데이팅앱
- 누적 다운로드 90만+ (런던, 뉴욕 중심)
- 시드 $3.5M 유치 (2021년 TechCrunch 보도)

**희소성 메카닉**
```
목요일 00:01 앱 오픈
→ 하루 최대 10명 매칭 제한
→ 저녁까지 대화 → 오프라인 만남 유도
→ 자정에 모든 채팅 삭제 → 다음 주 목요일까지 잠금
```

**설계 의도와 효과**
- **스와이프 피로 해소**: 일주일 치 에너지를 하루에 집중
- **의사결정 강제**: "지금 나가지 않으면 기회가 사라진다" → 만남 전환율 상승
- **오프라인 이벤트 연계**: 목요일 밤 런던 바 파티 → 앱 유저 오프라인 집결
- **실측 수치**: 단 하루에 좋아요 11만 건, 매칭 7,500건 발생

**왜 전체 성장에는 한계가 있나**
- 주 1회 제약이 일상적 앱 사용 습관 형성 방해 → 네트워크 효과가 느림
- 특정 도시(런던, 뉴욕)에 집중 → 지방 확장 어려움

**PO 교훈**: **희소성은 행동을 촉발하는 강력한 트리거**. "오늘 주선 가능" 토글, 또는 특정 요일/시간 소개 활성화 같은 제한적 희소성은 주선자 참여율 상승에 적용 가능.

### 1-7. Feeld (2014~현재 — 폴리아모리/비모노가미 커뮤니티)

**개요**
- 비모노가미, 폴리아모리, 개방적 관계를 위한 앱
- 7백만 유저, 성/젠더/관계 유형 20+ 선택지

**2024년 Constellation 기능**
```
본인 프로필에 최대 5명의 파트너/친구를 링크 가능
→ 연결된 사람들이 서로 인식 → "컨스텔레이션(별자리)" 구조
→ 그룹 채팅 + 관심사 기반 이벤트 발견
```

**PO 교훈**: 1대1 매칭 패러다임을 벗어나 **다자간 연결 구조를 앱 안에 명시적으로 지원**한 사례. 친구가 나와 함께 앱 안에 "연결된" 형태로 존재할 수 있음.

### 1-8. 아시아 데이팅앱 — Tapple, Pairs (일본), Tantan (중국), Azar (한국)

#### Tapple (CyberAgent, 일본)
- 누적 2,300만 유저
- **관심사 그룹 기반 매칭**: 취미 그룹 입장 → 같은 취미 사람들과 연결
- **24시간 "지금 나가고 싶어" 기능**: 즉흥 만남 유도 (Thursday와 유사한 희소성)
- **Solo Status Verification**: 공식 정부 문서 연동으로 싱글 확인 → 신뢰도 높음
- 리퍼럴 프로그램 관련 공개 데이터 없음

#### Pairs (Match Group Asia, 일본/대만)
- **Communities 기능**: 관심사 기반 그룹 → 커뮤니티 가입 후 자연스러운 연결
- **Ashiato(발자국)**: 내 프로필 방문자 추적 → 관심 있는 상대를 먼저 파악
- 공식 오프라인 이벤트 운영
- 친구 리퍼럴 공개 데이터 없음

#### Tantan (Hello Group, 중국)
- MAU 5,000만, 누적 매칭 40억+
- **Tribe 기능**: WeChat식 관심사 그룹 → 커뮤니티 → 만남
- Moments 피드 (WeChat과 유사)
- 2025년 "슬로우 데이팅" 방향으로 피봇 (스와이프 피로 대응)

#### Azar (HyperConnect/Match Group Asia, 한국)
- 누적 등록 4억명+, 2024년 미국 공식 출시
- 230개국, 19개 언어, 실시간 영상 통화 매칭
- **실시간 AI 번역**: 언어 장벽 없이 글로벌 연결
- 2024년 12월: Gen Z 고독 해소 앱으로 리포지셔닝
- 소셜 그래프 기반 친구 추천 기능 없음 — 완전 랜덤 연결 모델

**PO 교훈**: 일본/중국 앱의 공통점은 **커뮤니티→만남 퍼널**. 관심사 그룹에 먼저 들어가 소속감을 형성한 후 매칭으로 이어지는 구조가 신규 유저 진입 장벽을 낮춤.

### 1-9. Bumble BFF — 친구 찾기 기능 실패 사례

**개요**
- Bumble의 친구 매칭 사이드 기능 (2016년 출시)
- 2024년 별도 앱으로 분리 → 2025년 리뉴얼

**왜 실패했나**
- 데이팅 앱의 1:1 스와이프 매칭 패러다임을 **우정 형성에 그대로 적용** → 행동 목적이 달라서 맞지 않음
- 2023년 추정 매출 $1M — 수익화 실패
- CEO 발언: "We have been slow to realize this broader vision"

**반전 전략 (2024~2025)**
- 1:1 매칭 → 그룹/커뮤니티 패러다임으로 전환
- 커뮤니티 앱 Geneva 인수 (하지만 수익 $0 보고)
- 2025년 BFF 앱 리뉴얼: 그룹 연결, 커뮤니티 빌딩 중심으로 재설계

**PO 교훈**: **잘못된 UX 패러다임을 올바른 유저 니즈에 적용하면 아무리 수요가 있어도 제품이 실패한다**. 데이팅 매칭 ≠ 우정 형성. 각각의 소셜 행동에 맞는 별도 설계 필요.

### 2-1. Locket Widget (2021~현재)

**개요**
- 친구 사진을 아이폰 홈 스크린 위젯에 실시간 표시
- 누적 다운로드 8,000만+, DAU 900만+, 사진 공유 100억+
- 창업자 Matt Moss가 장거리 여자친구를 위해 만든 앱 → 제품-시장 적합성의 교과서

**핵심 바이럴 루프 구조**
```
[사용자 A가 앱 설치]
  → 베스트 프렌드 초대 (앱이 작동하려면 상대방도 설치해야 함)
  → 자동 생성 초대 문자 발송 ("OO님이 당신을 Locket에 초대했어요")
  → 상대방 앱 설치 필수 (위젯을 받으려면)
  → [사용자 B가 앱 설치] → 다시 자신의 친구 초대
  → 루프 반복
```

**"강제 동반 설치" 구조가 핵심**
- 위젯을 **받으려면** 상대방도 앱이 있어야 함 → 자연스럽게 설치 강제
- 유료 광고비 ≈ 0으로 출시 2주 만에 200만 가입
- TikTok 바이럴 (창업자 직접 영상 → 10만 뷰 → UGC 폭발)
- App Store 1위 달성 (2022년 1월)

**주요 수치**
- 총 다운로드 8,000만
- 시드 투자 $1,250만
- 2024년 흑자 전환
- 크리에이터 26명 → 게시물 누적 2억 9,800만 뷰, 공유 150만회

**PO 교훈**
1. **"앱이 작동하려면 친구를 초대해야 한다"** 구조가 가장 강력한 바이럴 루프
2. 초대 문자 자동 생성 → 사용자가 직접 카피라이팅할 필요 없음 → 마찰 제거
3. 개인적인 니즈(장거리 연애)를 해결하는 제품이 가장 강한 입소문 생성

2. 초대 문화가 사라진 후에도 유저가 남으려면, 매일 돌아올 이유가 있어야 함
3. 2개 초대장 = 각 유저가 2명의 직접 친구를 데려옴 → 신뢰 기반 성장

### 2-4. Discord (2015~현재)

**성장 메카닉**
- 서버 초대 링크 시스템: 무제한 사용, 영구 링크 → 공유하기 쉬움
- **친구 초대 → 서버 부스트 리워드** (테스트됨): 초대 수락 시 발신자에게 서버 부스트 제공
- 게임 커뮤니티 → e스포츠 팬 → 팬이 소셜 미디어에서 자신의 팀 Discord로 유입

**핵심 인사이트**
- 리워드 없이도 친구 초대가 일어나는 이유: **Discord 자체가 친구들과 있을 때만 가치 있는 앱**
- 서버 = 공간 → 공간을 공유하고 싶을 때 자연스럽게 초대 행동 발생
- 2016년: 가입 유저 2,500만 → 리퍼럴 마케팅이 Growth의 핵심

**PO 교훈**: **앱 내 가치가 "혼자서는 경험 불가"인 구조**가 가장 강력한 초대 동기. "친구가 없으면 소개를 볼 수 없는" 구조를 설계하면 초대가 자연스럽게 발생한다.

### 2-5. Duolingo Friend Streak (2023 출시)

**Friend Streak 핵심 수치**
- Friend Streak 1개 이상 있는 유저는 **일일 학습 완료율 22% 상승**
- Friend Streak 수가 늘어날수록 완료율 추가 상승
- 전체 유저의 57%가 친구 1명 이상 보유 (사회 기능 기반)

**바이럴 공유 트리거 (3가지)**

2. **황당한 문장**: 특이한 번역 문장이 밈이 됨 → 유머 콘텐츠로 공유
3. **리더보드 1~3위**: 주간 리그 상위권 → 경쟁심 + 증명 욕구 → 공유

**설계 결정 인사이트**

**PO 교훈**
1. 친구 기능은 리텐션 기능이기도 하다 — 연결된 유저는 혼자인 유저보다 22% 더 행동
2. **공유 트리거는 "성취" + "유머" + "경쟁"** — 이 중 하나를 충족하는 순간을 설계해야 함
3. 초대 퍼널의 초기 단계 UX를 반복 개선하는 것이 파워 유저 전환보다 임팩트가 크다

### 2-7. Venmo (2009~현재)

**소셜 피드가 핵심 바이럴 메카닉인 이유**

Venmo는 결제 앱이지만 **소셜 피드**가 성장의 핵심 드라이버였다.

- 거래 내역이 기본 공개 → 이모지와 한마디 설명이 소셜 피드에 노출
- 예시: "🍕🍺 for pizza and beer on Friday" → 비유저가 친구의 피드를 보면서 자연스럽게 인식
- "내 친구들이 쓰는 앱이다" 인식 → 설치 동기 유발
- 초기 타겟: 펜실베이니아대 학생 → 졸업 후 alumni 네트워크로 자연 확산

**성장 메카닉 분해**
```
각 거래 → 피드에 공개 (기본값)
→ 비유저 친구가 피드 확인 (링크 타고 앱 페이지 이동)
→ "친구들이 다 쓰네" 인식
→ 설치
→ 새 유저가 거래 → 피드 공개 → 루프 반복
```

**핵심 인사이트**: 유저가 의도적으로 초대하지 않아도, **앱 안의 행동 자체가 외부에 노출**되는 구조

**PO 교훈**: "소개 완료" 알림, "OO님이 당신을 주선해줬어요" 같은 외부 가시성 이벤트가 있으면 → 비유저 유입 루프 가능. 단, **프라이버시 기본값은 항상 opt-in**으로 설계해야 함.

### 2-8. Snapchat (2011~현재)

**친구 그래프가 코어 프로덕트인 이유**

**PO 교훈**: **앱 핵심 기능이 친구 없이는 작동하지 않는 구조** → 가장 강력한 초대 동기. 내친소 "소개 기능은 친구가 있어야 활성화"도 같은 원리.

### 2-9. Dropbox (2008~현재) — 리퍼럴 프로그램 교과서

**핵심 수치**

| 항목 | 수치 |
|---|---|
| 성장율 | 15개월 만에 10만 → 400만 유저 (+3,900%) |
| 바이럴 계수(K-factor) | 0.35 (매 10명이 3.5명 추가 유입) |
| 피크 기여율 | 일일 신규 가입의 35%가 리퍼럴에서 발생 |
| 2010년 4월 초대 발송 수 | 280만 건/월 |

**양면 보상 구조**
- 추천자 + 추천받은 사람 모두 500MB 스토리지 지급
- 최대 16GB까지 적립 가능

**핵심 설계 원칙**
1. **앱 내부 가치(스토리지)로 보상** — 현금이 아니라 제품 자체의 핵심 가치
2. **양면 보상** — 추천자만 받는 게 아니라 받는 사람도 혜택 → 죄책감 없이 초대 가능
3. **계단식 적립** — 최대 한도까지 계속 초대 동기 유지

**PO 교훈**: K-factor 0.35도 유료 광고 없이 39배 성장을 만들 수 있다. K > 1이 아니어도 충분히 의미 있다. 내친소에서 **씬구점으로 양면 보상**하는 아이디어의 근거.

### 4-1. K-factor (바이럴 계수)

```
K = 사용자당 평균 초대 수 × 초대→가입 전환율

K > 1.0  → 자생적 바이럴 성장
K 0.3~0.9 → 유료 채널 비용 30~90% 절감
```

| 앱 | K-factor | 결과 |
|---|---|---|
| Dropbox | ~0.35 | 15개월 +3,900% (10만 → 400만) |
| Locket Widget | 추정 >1.0 | 2주 200만 가입, 광고비 ≈ 0 |
| Clubhouse | 추정 >1.0 (단기) | 2개월 60만 → 1,000만 |
| BeReal | 추정 0.5~1.0 (초기) | 1년 내 7,000만 피크, 이후 급락 |

**데이팅앱에서 K > 1이 어려운 이유**:
매칭 성공 → 두 명 동시 이탈 → 전파자 풀이 지속적으로 축소됨. 주선자를 유지해야 하는 구조적 이유.

---

### 4-2. CAC (고객 획득 비용) 비교

| 채널 | CAC |
|---|---|
| 유료 광고 (데이팅앱) | $40~80 |
| 유기적 성장 | $1~5 |
| 리퍼럴 프로그램 | $5~15 |

## 5. PO 종합 인사이트

---

### 5-1. 성공 패턴 — 살아남은 기능의 공통점

**패턴 1: 앱이 작동하려면 친구가 필요한 구조**
- Locket: 위젯을 받으려면 상대방도 설치해야 함
- Snapchat: 스냅을 받을 사람이 없으면 앱이 의미 없음
- Discord: 서버가 있어야 하고, 서버에 사람이 있어야 가치 생김

**패턴 2: 주선자 역할에 지속 참여 이유 제공**
- Wingman: 리더보드 경쟁 + 역할 전환 자유
- Duolingo: Friend Streak → 상호 의존 관계 생성
- Strava: Kudos + 세그먼트 경쟁 → 꾸준한 참여 동기

**패턴 3: 소개 행동의 마찰을 최소화**
- Hinge Matchmaker: 아이스브레이커 자동 발송
- LinkedIn: 소개 이메일 템플릿 자동 생성
- Dropbox: 초대 링크 한 번 클릭으로 공유 가능

**패턴 4: 양면 보상 (추천자 + 추천받는 사람 모두 혜택)**
- Dropbox: 양측 500MB 지급
- BeReal: 친구 초대 → 두 사람 모두 콘텐츠 공유 가능
- 내친소 적용: 소개 성사 시 주선자 + 소개받는 사람 양측 씬구점

---

### 5-2. 실패 패턴 — 반복되는 실수

**실패 1: 주선자에게 남을 이유가 없음 (Ship 실패 원인)**
- 친구 매칭이 성공해도 주선자에게 돌아오는 것이 없었음
- 역할이 분리되어 있어 지속 사용 동기 없음

**실패 2: 잘못된 UX 패러다임 적용 (Bumble BFF 실패 원인)**
- 데이팅 1:1 스와이프를 우정 형성에 그대로 적용
- 우정은 그룹/커뮤니티 패러다임이 맞음

**실패 3: 초기 바이럴만 있고 핵심 가치 없음 (Clubhouse, BeReal 하락 원인)**
- 초대 희소성·인증 효과로 급성장 → 그 효과가 사라지면 MAU 70% 이탈
- 반드시 바이럴과 핵심 가치 루프가 연결되어야 함

**실패 4: 프라이버시 침해 (Hinge Matchmaker 2017 "creepy" 반응)**
- 사용자 동의 없이 소셜 그래프 노출 → 불쾌감
- BuzzFeed 기사 제목: "Hinge Launches A Matchmaking App, And It's A Little Creepy"
- **동의 없는 프로필 공유는 신뢰를 파괴함 → 양쪽 Opt-in 필수**

---

### 5-3. 내친소를 위한 직접 적용 원칙

| 원칙 | 근거 앱 | 내친소 적용 |
|---|---|---|
| 소개 기능은 주선자 역할과 소개받기 역할을 통합 | Wingman | 피드 내에서 역할 전환 자유 |
| 앱이 작동하려면 친구가 있어야 하는 구조 | Locket, Snapchat | "소개 보내기"는 상대방도 내친소 가입 시 활성화 |
| 주선자에게 지속 참여 동기 제공 | Wingman, Duolingo | 소개 성사 시 씬구점 + 리더보드 |
| 소개 행동의 마찰 최소화 | Hinge, LinkedIn | 추천사 템플릿 제공 + 한 번 탭으로 전달 |
| 양면 보상 | Dropbox | 주선자 + 소개받는 사람 모두 씬구점 |
| 공통 친구를 프로필 최상단에 표시 | Cerca, LinkedIn | 공통 친구 뱃지 → 신뢰 신호 |
| 프라이버시는 기본값 opt-in | Hinge Matchmaker 실패 교훈 | 친구가 허락한 경우에만 소개 가능 |
| 희소성으로 참여율 상승 | Thursday | "오늘 주선 가능" 토글 |
