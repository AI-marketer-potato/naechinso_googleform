## 1. Cerca 심층 분석 — 내친소 핵심 레퍼런스

> **왜 Cerca가 내친소의 핵심 레퍼런스인가:** Cerca는 "낯선 사람이 아닌, 아는 사람의 아는 사람과 만난다"는 철학을 가진 유일한 글로벌 데이팅앱이다. 내친소의 "친구가 추천하는 사람과 만난다"와 동일한 방향성을 가지되, 접근 방식(자동화 vs 능동적 주선)이 다르다. Cerca의 성공 요소와 한계를 깊이 이해하는 것이 내친소 설계의 출발점이다.

### 1-5. 제품 설계 심층 분석

#### 핵심 컨셉: "Date Mutuals, Not Strangers"
- 연락처(phone contacts)를 동기화하면, 1촌(직접 아는 사람) 또는 2촌(친구의 친구)만 프로필로 노출
- 완전한 낯선 사람은 **절대 보여주지 않음**
- 각 프로필에 **공통 친구 이름이 실명으로 표시**됨

#### 연락처 기반 1촌/2촌 매칭 시스템
```
1촌: 내 연락처에 있는 사람 (직접 아는 사이)
2촌: 내 연락처에 있는 누군가의 연락처에 있는 사람 (친구의 친구)

→ 연락처 동기화는 필수 — 전체 매칭 시스템의 기반
→ 유저가 많은 연락처를 가지고 있을수록, 그리고 그 연락처 사람들이
   Cerca에 가입할수록 더 많은 매칭 풀이 생김
→ 사용자가 차단할 사람을 직접 선택 가능 (가족, 전 연인 등)
```

#### 매일 4명 추천 시스템
- **왜 4명인가:** Slayton — "1분에 100개 프로필을 보는 세상은 있어서는 안 된다. 각 프로필에 대해 진지하게 생각해야 한다"
- 스와이프 피로감(swipe fatigue) 원천 차단
- 선택 과부하(choice paralysis) 방지
- 각 프로필에 더 많은 시간과 주의를 투자하도록 유도

#### 오후 8시 매칭 공개 시스템
- 매일 동부시간 오후 8시(8PM EST)에 새로운 매치가 공개됨
- **데일리 이벤트 효과**: 매일 저녁 8시에 앱을 확인하는 습관 형성 (appointment viewing)
- 업계 평균 데이팅앱 Day 30 리텐션 5~6%인 상황에서, 시간 기반 메커니즘은 차별화된 리텐션 전략

**Thursday와의 핵심 차이:**

| 구분 | Thursday | Cerca |
|------|----------|-------|
| 제한 방식 | 목요일 하루만 앱 사용 (주 1회) | 매일 작동, 하루 1회 특정 시간에 결과 공개 |
| 희소성 수준 | 극단적 (6일간 앱 사망) | 일상적 (매일 이벤트, 적절한 제한) |
| 습관 형성 | 실패 (주 1회는 습관이 안 됨) | 성공 (매일 저녁 8시 = 루틴) |
| 최종 결과 | 2025년 앱 셧다운 | 지속 성장 중 |

#### 프로필 공개 순서 설계
```
1단계: 공통 친구 이름 (실명) — "누구를 통해 연결되는가"
2단계: 배경 정보 (학교, 직업 등)
3단계: 사진

→ 외모보다 "연결의 맥락"을 우선시하는 설계
→ S'More의 블러 리빌과 유사하지만, 관계 맥락으로 시작한다는 점이 차별화
```

#### 스와이프 없는 UX — 익명 좋아요
- 좋아요(Like)는 **완전히 익명** — 상대가 나를 좋아하지 않으면 절대 알 수 없음
- 좋아요를 받으면 알고리즘이 해당 프로필을 상대 피드에 부스팅
- Slayton: "사람들은 거절을 매우 두려워한다" → 거절의 두려움 제거
- 양쪽 모두 좋아요를 누르면 매치 성립

### 1-6. 성장 전략 심층 분석

#### 캠퍼스 우선 전략 (Campus-First)
- Georgetown University에서 시작 → 학부생 1/3이 가입하는 높은 침투율 달성
- 대학 캠퍼스는 **밀도 높은 소셜 네트워크** — 연락처 기반 매칭의 최적 환경
- Tinder, Facebook 모두 대학 캠퍼스에서 시작한 것과 동일한 전략
- Georgetown → USC → UPenn → Duke → Emory 순서로 자연 확산
- NYC를 중심 허브로 설정 (대학 졸업생들이 모이는 도시)

#### 바이럴 루프 구조 — "구조적 바이럴"
```
[유저 A가 가입]
→ 연락처 동기화
→ 매칭 풀이 제한적 (주변에 Cerca 유저가 적으면)
→ 매칭 풀을 넓히려면 친구들이 가입해야 함
→ 자발적 초대 발생 (내재적 인센티브)
→ [유저 B가 가입] → 유저 A의 매칭 풀 확대 + 유저 B도 같은 루프
→ 반복
```
- 핵심: 유료 보상이 아닌 **제품 가치 자체가 초대 동기** (Dropbox 모델과 동일)
- 2026년 3월 업데이트(v21.0.4)에서 리퍼럴 시스템 개편

#### 캠퍼스 앰배서더 프로그램
- 각 대학에 앰배서더를 모집 (OnCampusNation 통해 공식 모집)
- 주급 지급 (Zelle, Venmo, Apple Pay)
- 소셜 미디어, 입소문, 캠퍼스 이벤트로 홍보
- Bumble/Tinder의 캠퍼스 앰배서더 전략을 벤치마킹

#### IRL 이벤트
- 공통 친구와 함께하는 파티, 캠퍼스 활성화 이벤트
- 시드 펀딩으로 오프라인 이벤트 확대 진행 중
- AI 기능 도입은 의도적으로 배제 — "진짜 사람 간의 연결"이라는 브랜드 정체성 유지

#### 여성 비율 60%+ — 데이팅앱 업계 최대 이변

| 앱 | 남:여 비율 |
|----|-----------|
| Tinder | 약 75:25 (남성 압도적) |
| Bumble | 약 60:40 (남성 다수) |
| Hinge | 약 65:35 (남성 다수) |
| **Cerca** | **약 40:60 (여성 다수)** |

- 대부분의 데이팅앱은 남성 비중이 훨씬 높은 상황에서, **여성이 더 많은 것은 극히 이례적**
- 안전 중심 설계("아는 사람의 아는 사람") + 공통 친구 실명 표시가 여성 유저 신뢰를 확보한 결과
- 데이팅앱에서 여성 유저 확보는 "닭과 달걀" 문제의 핵심이므로, 이 지표는 **Cerca의 가장 강력한 경쟁력**

#### 주요 사용자 피드백
- **긍정:** 뮤추얼 기반 매칭이 대화의 질을 높인다, 기존 앱보다 의미 있는 연결 느낌
- **부정:** 전체 연락처 동기화에 대한 프라이버시 우려 (가족/전 연인이 내 데이팅앱 사용을 알 수 있음)

#### "소셜 그래프 데이팅" 카테고리의 다른 플레이어
- **Facebook Dating:** 페이스북 소셜 그래프 활용, 공통 친구 기반. 그러나 페이스북 자체의 Gen Z 이탈로 한계

- Cerca의 구체적 수익 모델은 **아직 공개되지 않음** (피치덱에서도 일부 삭제됨)
- 현재 무료 앱으로 운영 중
- Corazon Capital의 Phil Schwarz가 **Tinder 구독 모델 도입을 주도한 인물**이므로, 향후 프리미엄 구독 모델 도입이 유력

**가능한 수익 모델 방향:**

| 모델 | 설명 | 가능성 |
|------|------|--------|
| 추가 프로필 열람 | 일일 4명 이상 프로필 열람 | 높음 |
| 누가 나를 좋아했는지 보기 | Hinge/Tinder 패턴 | 높음 |
| 슈퍼라이크/우선 노출 | 특정 프로필에 강한 관심 표현 | 중간 |
| IRL 이벤트 티켓 | 오프라인 이벤트 판매 | 중간 |
| 프리미엄 구독 | 전체 기능 번들 | 높음 |

### 1-11. 내친소가 Cerca에서 배울 핵심 5가지

1. **일일 제한(4명) + 시간 이벤트(오후 8시) = 습관 형성의 핵심 조합.** 내친소도 "매일 저녁 9시, 오늘의 추천 공개" 같은 데일리 이벤트를 도입하면 DAU 리텐션에 직접 기여

2. **프로필 공개 순서: 관계 맥락 먼저.** 공통 친구 → 배경 → 사진 순서. 내친소에서는 "추천인의 한마디 먼저 → 성격/가치관 → 사진" 순서로 변환 가능

3. **여성 60%+ 달성 전략.** "아는 사람의 아는 사람"이라는 안전감이 여성 유저를 끌어들임. 내친소의 "친구가 추천하는 사람" 구조도 동일한 안전감 제공 → 마케팅에서 여성 타겟에 강하게 어필

4. **캠퍼스 앰배서더 프로그램.** 한국 대학 캠퍼스 단위로 밀도 높은 유저 풀을 먼저 만드는 전략. 연세대, 고려대, 서울대 등 캠퍼스별 침투율 목표 설정

5. **내재적 바이럴 루프.** "친구가 가입하면 나의 매칭 풀이 넓어진다"는 구조적 인센티브. 내친소에서는 "친구가 추천해줘야 소개받을 수 있다" → 자연스러운 초대 동기

### 1-12. Cerca의 한계 & 내친소 차별화 기회

| Cerca의 약점 | 내친소가 강한 이유 |
|-------------|-------------------|
| **프라이버시 문제** — 전체 연락처 동기화에 대한 강한 거부감 | 추천인 기반이므로 연락처 전체 공개 불필요. 범위가 제한적 |
| **수동적 소셜 그래프** — 기계적으로 연락처를 매칭할 뿐, "이 사람이 너한테 잘 맞을 것 같아"라는 인간적 큐레이션 없음 | 친구가 직접 "이 사람 너한테 어울려"라고 판단하는 **능동적 주선** |
| **추천사/보증 부재** — 친구가 상대에 대해 써주는 추천글 없음 | 추천인의 한마디가 프로필의 핵심 요소. 훨씬 풍부한 정보 제공 |
| **콜드 스타트** — 주변에 앱 유저가 없으면 매칭이 0 | 추천인 한 명만 있어도 시작 가능 |
| **깊이 부족** — 넓은 소셜 그래프를 자동으로 훑지만 깊이가 얕음 | 소수지만 깊이 있는 추천이 가능 |

### 3-1. Ship (Match Group x Betches Media, 2019~2022)

**개요**
- Match Group과 미디어 브랜드 Betches가 공동 개발한 크루 기반 매칭 앱
- 2019년 1월 iOS 출시, Match Group 패밀리 중 가장 빠르게 성장한 앱으로 시작
- 2022년 9월 30일 서비스 종료

**핵심 메커니즘**
- **Crew 시스템**: 사용자가 친구를 "크루(crew)"에 초대 → 크루 멤버가 당사자를 위해 스와이프 대리 수행
- **Crew Chat**: 잠재적 매칭 상대를 두고 그룹 채팅으로 의견 나눔
- **결과**: 전체 매칭의 53%가 크루의 대리 스와이프에서 발생 — 핵심 기능은 기술적으로 작동함

**왜 실패했나 — PO 관점**

| 원인 | 설명 |
|---|---|
| 주선자 리텐션 제로 | 친구 매칭이 성공하면 당사자는 앱을 떠나지만, 주선자는 계속 써야 하는 이유가 없었다 |
| 역할 분리 문제 | 주선자 역할과 소개받는 역할이 분리되어 있어, 이미 연애 중인 사람이 쓸 이유가 없었다 |
| 보상 부재 | 성사시켜줘도 주선자에게 돌아오는 인센티브가 없었다 |
| 브랜드 정체성 약화 | Match Group의 다른 앱과 차별화 포인트가 시간이 지나면서 희석됨 |

**CEO 발언 (Bernard Kim, 2022)**: "We must focus our investment on brands that are growing and pull back in areas that are not driving growth."

**PO 교훈**: 친구 소셜 기능이 기술적으로 잘 작동해도, **주선자가 계속 써야 할 이유**가 없으면 앱이 유지되지 않는다. 역할 통합이 선행 조건.

### 3-2. Wingman (2018~현재 운영 중)

**개요**
- 친구가 대신 프로필을 작성하고 매칭하는 앱
- 자신의 소개도 받으면서 친구의 소개도 도와주는 이중 구조

**핵심 메커니즘 — Dual Mode**
```
Wingee (소개받는 사람) 역할:
  → 마음에 드는 카드 발견 → 오른쪽 스와이프
  → Wingman에게 nudge(연락 요청)
  → Wingman이 소개 진행

Wingman (주선자) 역할:
  → 친구 프로필 직접 작성 ("good/bad/ugly" 항목으로 솔직하게)
  → 친구를 위해 스와이프
  → 소개 성사 시 리더보드 포인트 획득
```

**핵심 UX 결정**
- 프로필을 **친구가 직접 작성** → 자기 PR보다 신뢰도 높음 (Wingman 내부 조사 기준 80% 높은 매칭률 주장)
- **복수 Wingman 허용** → 여러 친구가 동시에 주선 경쟁 → 리더보드 경쟁 생성
- 역할 전환 자유 → 오늘은 주선자, 내일은 소개받는 쪽 → 이탈 방지

**Ship과의 결정적 차이**

| 구분 | Ship | Wingman |
|---|---|---|
| 역할 구조 | 당사자 + 크루 분리 | 하나의 앱에서 역할 전환 |
| 주선자 보상 | 없음 | 리더보드 포인트 |
| 지속 생존 | 2022년 종료 | 현재까지 운영 |

**PO 교훈**: Wingman이 Ship보다 오래 살아남은 이유는 **역할 통합과 주선자 리워드**. 이 두 가지가 없으면 주선 기능은 유지되지 않는다.

### 3-3. Hinge Matchmaker (2017 — 단독 앱, 이후 통합)

**개요**
- Hinge가 2017년 9월 출시한 별도 스탠드얼론 앱
- "연애 중인 사람도 싱글 친구를 위해 데이팅앱에 참여할 수 있다"는 컨셉

**UX Flow (3단계)**
```
Step 1: 내 Facebook 친구 중 Hinge 사용자를 확인
Step 2: 두 명을 지정 → 매칭 추천
Step 3: 두 사람 모두에게 아이스브레이커 메시지 자동 전송 → 연결 성사
```

**핵심 설계 원칙**
- **Facebook 소셜 그래프 활용**: 이미 아는 사람들 사이에서 연결 → 신뢰도 확보
- **아이스브레이커 자동 발송**: 소개받는 사람이 어색함 없이 대화 시작 가능
- **초대 링크 흐름**: 비사용자에게 초대 링크 발송 → 신규 유저 유입 루프 생성

**한계와 이후 변화**
- 스탠드얼론 앱이 메인 앱과 따로 존재 → 사용자 분산 문제
- 이후 Hinge는 메인 앱 내 "프로필 공유(Send Profile)" 기능으로 흡수
- 2025년 2월에는 "Match Note" (매칭 상대에게 개인 정보 비공개 공유) 기능 출시

**PO 교훈**: 아이스브레이커 자동 발송이 핵심 마찰(어떻게 말 걸지)을 제거했다. **소개 행동의 마찰을 낮추는 것이 전환율 결정**.

### 3-4. S'More (2019~현재) — 슬로우 리빌 데이팅 심층 분석

**개요**
- **"Love Is Blind" 앱 버전** — 외모 블러 후 대화로 점진적 언블러
- 창업자 Adam Cohen-Aslatei가 2019년 런칭
- 핵심 철학: "외모보다 내면을 먼저 알아가자"
- 앱스토어 슬로건: "Anti-Superficial Dating"

**슬로우 리빌 메카닉 심층 분석**
```
1단계: 매칭 → 사진은 완전 블러 처리 상태
2단계: 채팅 → 대화할수록 사진 조각씩 언블러
3단계: 아이콘 상호 작용("wink") → 추가 블러 해제
4단계: 소셜 미디어 피드도 언블러 후 공개

→ "대화의 깊이"가 외모 공개의 조건
→ 외모 우선 판단을 구조적으로 차단
```

**비디오 스피드데이팅 기능 (2020년 추가)**
- 코로나 팬데믹 대응으로 라이브 비디오 스피드데이팅 도입
- 프로필 블러 상태에서 짧은 영상 대화 → 대화 후 블러 해제 여부 결정
- 앱 내 콘텐츠 요소: 셀럽 데이팅 콘텐츠, 인터뷰, 데이팅 팁

**주요 수치**
- 프로필 상호작용의 35% 이상이 셀럽 데이팅 콘텐츠 시청 후 발생
- 2021년 멤버십 2배 증가
- 2022년 시리즈 A 투자 유치 (금액 비공개)
- 뉴욕, LA, 시카고, 보스턴 등 미국 주요 도시에서 운영

**S'More의 한계**
- 블러 해제까지의 과정이 일부 유저에게 지나치게 느리게 느껴짐
- "결국 외모는 봐야 한다"는 현실과 "외모를 가린다"는 설계 사이의 긴장
- 주류 앱(Hinge, Bumble)에 비해 유저 풀이 제한적 → 매칭 확률 저하
- 슬로우 리빌 컨셉의 카피캣 등장 (Bumble도 유사 기능 테스트)

**Cerca/내친소와의 비교:**

| 접근 | S'More | Cerca | 내친소 |
|------|--------|-------|--------|
| 외모 가리기 | 블러 처리 | 공통 친구 먼저 표시 | 추천사 먼저 표시 |
| 정보 공개 순서 | 대화량 기반 | 관계 맥락 기반 | 추천인 신뢰 기반 |
| 목적 | 외모 편향 제거 | 신뢰 형성 | 신뢰 + 인간적 보증 |

**PO 교훈**: 외모 우선 노출이 여성 이탈의 주요 원인 중 하나. **프로필 노출 순서를 조정하는 것만으로도 매칭 품질과 여성 리텐션이 달라질 수 있다**. S'More는 "블러"로 가렸고, Cerca는 "공통 친구"를 먼저 보여줬고, 내친소는 "추천사"를 먼저 보여주는 것이 이 같은 맥락의 접근.

### 3-5. Coffee Meets Bagel (2012~현재) — 큐레이션 데이팅의 원조

**개요**
- 한국계 미국인 Kang 자매 3명(Arum, Dawoon, Soo)이 2012년 창업
- 2015년 Shark Tank에서 Mark Cuban의 $30M 인수 제안을 거절한 것으로 유명
- 핵심 철학: "Quality over Quantity" — 매일 정오에 엄선된 소수의 프로필만 제공
- 누적 $1.5억+ 펀딩

**핵심 메카닉 — "정오의 배글"**
```
매일 정오 12시 → 여성에게 "Bagel" (큐레이션된 매칭 프로필) 제공
  → 여성이 먼저 수락/거절
  → 수락한 프로필만 남성에게 공개
  → 24시간 내 대화 시작 안 하면 매칭 소멸

추가 발견 기능: "Discover" — 추가 프로필 열람 (유료 "Beans" 필요)
```

**Cerca/내친소와의 관련성:**

| 측면 | Coffee Meets Bagel | Cerca | 내친소 |
|------|-------------------|-------|--------|
| 일일 제한 | 매일 정오 6명 | 매일 오후 8시 4명 | 하루 3회 (예정) |
| 시간 이벤트 | 정오 (점심시간) | 오후 8시 (저녁시간) | 저녁 9시 (예정) |
| 매칭 소멸 | 24시간 | 미공개 | 72시간 (예정) |
| 큐레이션 주체 | 알고리즘 | 소셜 그래프 | 친구 (인간) |

**현재 상태 (2025~2026)**
- 여전히 운영 중이나 Hinge에 시장 주도권을 많이 빼앗김
- "Beans" 가상화폐 기반 수익 모델 유지
- AI 매칭 알고리즘 고도화 (프로필 선호도 학습)
- 2024년 CMB Premium 구독 출시 ($35/월)
- 아시아계 미국인 커뮤니티에서 여전히 강한 브랜드

**소셜 기능 현황:**
- "CMB for Friends" — 친구에게 자신의 매칭 프로필을 공유하고 의견을 구하는 기능 (제한적)
- Hinge 스타일의 "Send Profile" 기능 보유
- 그러나 Cerca나 내친소처럼 소셜 그래프를 핵심 매칭 메커니즘으로 사용하지는 않음

**CMB의 교훈 — "정오" 시간 이벤트의 증명:**
- CMB가 "매일 정오"에 매칭을 제공한 것은 Cerca의 "매일 오후 8시"보다 먼저 나온 시간 이벤트 모델
- 이 모델이 10년 넘게 유지되고 있다는 사실 자체가 **일일 시간 이벤트의 지속 가능성을 증명**
- 내친소의 "매일 저녁 9시 추천 공개" 모델의 강력한 선례

**PO 교훈**: CMB는 "큐레이션 + 시간 제한"의 조합이 10년 넘게 작동함을 증명했다. 그러나 **큐레이션 주체가 알고리즘인 것이 한계** — 내친소는 큐레이션 주체가 "친구"이므로 신뢰도가 훨씬 높다.

### 3-6. Thursday (영국, 2021~2025) — 희소성 실험의 교과서적 실패

**개요**
- "목요일 하루만 활성화"되는 데이팅앱
- 창업자: George Rawlings
- 누적 유저 200만+ (런던, 뉴욕 중심)
- 총 펀딩 $3.62M

**희소성 메카닉**
```
목요일 00:01 앱 오픈
→ 하루 최대 10명 매칭 제한
→ 저녁까지 대화 → 오프라인 만남 유도
→ 자정에 모든 채팅 삭제 → 다음 주 목요일까지 잠금
```

**전성기 수치**
- 단 하루에 좋아요 **110,000건**, 매칭 **7,500건** 발생
- 누적 수천 건의 데이트, 100건 이상의 결혼, 29명의 아기
- 전 세계 **60개 도시**에서 오프라인 이벤트 운영
- Guinness 등 브랜드와 협업 이벤트
- 대담한 옥외 광고, 바이럴 소셜미디어 콘텐츠, 밈 마케팅

**2025년 1월: 앱 공식 셧다운**

| 항목 | 내용 |
|------|------|
| **셧다운 시점** | 2025년 1월 |
| **창업자 발언** | George Rawlings: "데이팅앱은 죽어가고 있다" |
| **셧다운 사유** | "급격히 감소하는 소비자 관심" |
| **피봇 방향** | **Thursday Events** — 오프라인 싱글 이벤트 티켓 판매 사업으로 완전 전환 |
| **추가 사업** | 여행 부문, 의류 라인, 신규 국가 확장 계획 |
| **핵심 교훈** | 앱 자체보다 오프라인 이벤트가 더 지속 가능한 비즈니스 |

**왜 실패했는가 — 구조적 분석**

| 문제 | 설명 |
|------|------|
| **습관 형성 불가** | 주 1회는 일상적 앱 사용 습관이 되지 않음. 매주 목요일을 기억하는 것 자체가 부담 |
| **6일간 앱 사망** | 목~수요일까지 앱이 완전히 비활성 → 지속적 인게이지먼트 불가 |
| **관계 연속성 부족** | 자정에 매칭 소멸 → 대화가 깊어질 시간이 없음 |
| **도시 집중** | 런던, 뉴욕에 집중 → 지방/중소도시 확장 어려움 |
| **네트워크 효과 느림** | 주 1회 제약이 유저 간 상호작용을 제한 → 바이럴 루프가 약함 |

**Thursday vs Cerca: "희소성" 설계의 성패를 가른 차이**

| 비교 항목 | Thursday (실패) | Cerca (성공) |
|-----------|----------------|-------------|
| 제한 단위 | 요일 (주 1회) | 시간 (매일 1회) |
| 앱 활성 빈도 | 주 1일 | 매일 |
| 습관 형성 | 불가 (7일 간격) | 가능 (24시간 간격) |
| 매칭 소멸 | 자정에 완전 삭제 | 소멸 없음 |
| 최종 결과 | 2025년 앱 셧다운 | 지속 성장 중 |

**Thursday Events — 피봇 후 현재 상태:**
- 오프라인 싱글 이벤트를 60개 도시에서 계속 운영
- 이벤트 티켓 판매가 주 수익원
- 앱은 죽었지만 이벤트 브랜드는 살아남음
- 이는 **오프라인 이벤트 자체의 가치**를 역설적으로 증명

**PO 교훈 3가지:**

1. **"시간 제한"은 써야 하지만 "요일 제한"은 위험**: Thursday처럼 특정 요일에만 앱을 여는 것은 실패했지만, Cerca/CMB처럼 "매일 특정 시간에 결과 공개"는 성공적. 내친소에서는 **매일 오후 9시 추천 공개 + 72시간 응답 기한** 같은 구조가 적절

2. **오프라인 이벤트는 별도 수익원이자 온보딩 채널**: Thursday가 앱을 접고도 이벤트는 유지한다는 사실이 시사하는 바가 큼. 내친소도 "친구가 친구를 소개하는 오프라인 밋업"을 분기별로 개최하면, 신규 유저 온보딩 + 브랜드 강화 + 추가 수익원이 가능

3. **매칭 소멸은 부드럽게**: 자정 소멸은 긴박감을 주었지만 관계 발전을 방해. "72시간 내 대화 시작하지 않으면 매칭 종료" 정도의 완화된 버전이 적합

### 3-7. Raya (2015~현재) — 초대 전용 독점 데이팅의 운영 전략

**개요**
- "데이팅앱의 Soho House" — 초대 전용(invite-only) 독점 데이팅 & 소셜 네트워크 앱
- 창업자: Daniel Gendelman (2015년)
- 주요 유저: 연예인, 인플루언서, 크리에이터, 기업가 등 "크리에이티브 클래스"
- 구독료: 월 $7.99 (데이팅) 또는 $9.99 (데이팅 + 소셜)

**독점성 유지 메커니즘**

```
가입 프로세스:
1. 기존 멤버의 추천 필수
2. 신청서 작성 (직업, Instagram 팔로워 수, 관심사 등)
3. 알고리즘 + 위원회 심사 (수용률 약 8~10%)
4. 대기자 명단 (수만~수십만 명이 대기)
5. 위반 시 즉시 퇴출 (스크린샷 촬영 시 경고 등)
```

**독점성이 작동하는 이유:**

| 메커니즘 | 설명 |
|----------|------|
| **사회적 희소성** | "나는 Raya 멤버다" 자체가 사회적 신호 → 가입 욕구 유발 |
| **추천인 책임** | 추천한 사람이 문제를 일으키면 추천인에게도 불이익 → 품질 자정 |
| **스크린샷 금지** | 앱 내 콘텐츠 유출 방지 → 프라이버시 보장 → 유명인 안심 |
| **낮은 수용률** | 8~10% 수용률이 FOMO를 극대화 |
| **구독 필수** | 무료 유저 없음 → 진지한 유저만 남음 |

**Raya의 소셜 기능:**
- 데이팅 + 소셜 네트워킹 통합 (직업 네트워킹 기능 보유)
- 이벤트 기능: Raya 멤버 전용 오프라인 이벤트
- "Creative Community" 포지셔닝으로 데이팅 외 가치 제공

**Raya가 장기 생존하는 이유:**
- 독점성 자체가 제품의 핵심 가치이자 바이럴 메커니즘
- 유명인 유저가 PR 효과를 자동으로 생성 (뉴스에 자주 언급)
- 구독 기반 안정 수익
- 프라이버시 보호가 고가치 유저를 유지

**내친소 적용 가능성:**

| Raya 요소 | 내친소 적용 | 적용 방식 |
|-----------|-----------|-----------|
| 추천인 책임 | 적용 가능 | 추천인이 보증한 사람이 문제 시 추천인 신뢰도 하락 |
| 심사 프로세스 | 부분 적용 | 친구 추천 = 자연스러운 1차 스크리닝 |
| 대기자 명단 | 적용 가능 | 런칭 시 대기열 + 친구 초대 시 순위 상승 (Robinhood 모델) |
| 스크린샷 금지 | 검토 필요 | 프로필 보호 차원에서 유사 기능 고려 가능 |

**PO 교훈**: Raya의 "추천인 책임제"는 내친소의 핵심 컨셉과 직접 연결된다. **추천인이 보증한 사람의 품질이 추천인의 신뢰도에 영향을 주는 구조**를 만들면, 추천의 품질이 자연스럽게 올라간다. 또한, 초기 런칭 시 Raya식 대기열 + Robinhood식 순위 상승을 결합하면 강력한 바이럴 효과 기대.

### 3-8. Locket Widget (2021~현재) — "강제 동반 설치"의 교과서

**개요**
- 친구 사진을 아이폰 홈 스크린 위젯에 실시간 표시
- 창업자 Matt Moss가 장거리 여자친구를 위해 만든 앱 → 제품-시장 적합성의 교과서
- 핵심: **"앱이 작동하려면 상대방도 설치해야 함"** → 가장 강력한 바이럴 구조

**핵심 수치**

| 지표 | 수치 |
|------|------|
| 총 다운로드 | 8,000만+ |
| DAU | 900만+ |
| 사진 공유 | 100억+ |
| 시드 투자 | $1,250만 |
| 출시 2주 만 가입 | 200만 (유료 광고비 ≈ 0) |
| 크리에이터 콘텐츠 | 26명 → 게시물 2억 9,800만 뷰, 공유 150만회 |
| 흑자 전환 | 2024년 |

**바이럴 루프 구조**
```
[사용자 A가 앱 설치]
  → 베스트 프렌드 초대 (앱이 작동하려면 상대방도 설치해야 함)
  → 자동 생성 초대 문자 발송 ("OO님이 당신을 Locket에 초대했어요")
  → 상대방 앱 설치 필수 (위젯을 받으려면)
  → [사용자 B가 앱 설치] → 다시 자신의 친구 초대
  → 루프 반복
```

**성장 요인 분석:**
- **강제 동반 설치**: 위젯을 받으려면 상대방도 앱이 있어야 함 → K-factor > 1.0
- **TikTok 바이럴**: 창업자 직접 영상 → 10만 뷰 → UGC 폭발
- **App Store 1위** 달성 (2022년 1월)
- 개인적인 니즈(장거리 연애)를 해결하는 제품이 가장 강한 입소문 생성

**Cerca/내친소와의 연결:**
- Cerca도 "연락처에 유저가 있어야 매칭 풀이 생김" → 유사한 구조적 바이럴
- 내친소도 "친구가 추천해줘야 소개받을 수 있음" → 동일한 **강제 동반 사용** 구조
- Locket의 "초대 문자 자동 생성"을 내친소의 "카카오톡 공유 카드"로 변환 가능

**PO 교훈**:
1. **"앱이 작동하려면 친구를 초대해야 한다"** 구조가 가장 강력한 바이럴 루프
2. 초대 문자 자동 생성 → 사용자가 직접 카피라이팅할 필요 없음 → 마찰 제거
3. 내친소의 "친구 추천이 있어야 소개받을 수 있다" = Locket의 "상대방도 설치해야 위젯이 작동한다"와 동일 원리

**바이럴 성장 메카닉**

- 알림 동시성 → 모든 친구가 같은 순간 알림 받음 → "나도 찍어야 해" FOMO 유발

**왜 급락했나**

- 경쟁사 카피캣: Instagram "Candid Stories", Snapchat 듀얼 카메라 → 차별점 소멸

**PO 교훈**
1. **초기 바이럴은 설계 가능하지만, 지속 리텐션은 핵심 가치에 달려 있다**
2. 친구 초대 강제 구조는 성장을 폭발시키지만 → 앱 내 가치가 없으면 유지 불가
3. 캠퍼스 앰배서더 프로그램은 특정 타겟 커뮤니티(대학생) 공략에 효과적

### 4-2. Clubhouse (2020~2021 급성장, 이후 하락)

**초대 전용 성장 메카닉**

| 기간 | 지표 |
|---|---|
| 2020년 12월 | 60만 유저 |
| 2021년 2월 | 1,000만 유저 (2개월 만에 16배) |
| 2021년 4월 | 월간 다운로드 900만으로 급락 (2월 대비 91% 감소) |

**초대 시스템 설계**
- 기존 유저에게 **초대장 2장만 지급** → 초대장이 희소 자원이 됨
- eBay, Craigslist에서 초대장 실제 거래 발생 → FOMO 극대화
- 셀럽/VC/기업인 초기 유저 → 한 번 들어간 사람이 그 가치를 외부에 전파

**한계**
- 앱 오픈 후 1개월 내 MAU 70% 이탈
- 초대 희소성이 사라지자 → 앱의 핵심 가치 혼자서는 유지 불가
- Android 출시 지연 → iOS 유저 기다리는 동안 경쟁자(Twitter Spaces, Discord) 성장

**PO 교훈**
1. **초대 희소성은 강력한 초기 성장 도구**지만 앱의 핵심 가치가 아님
2. 초대 문화가 사라진 후에도 유저가 남으려면, 매일 돌아올 이유가 있어야 함
3. 2개 초대장 = 각 유저가 2명의 직접 친구를 데려옴 → 신뢰 기반 성장

### 4-3. Discord (2015~현재)

**성장 메카닉**
- 서버 초대 링크 시스템: 무제한 사용, 영구 링크 → 공유하기 쉬움
- **친구 초대 → 서버 부스트 리워드** (테스트됨): 초대 수락 시 발신자에게 서버 부스트 제공
- 게임 커뮤니티 → e스포츠 팬 → 팬이 소셜 미디어에서 자신의 팀 Discord로 유입

**핵심 인사이트**
- 리워드 없이도 친구 초대가 일어나는 이유: **Discord 자체가 친구들과 있을 때만 가치 있는 앱**
- 서버 = 공간 → 공간을 공유하고 싶을 때 자연스럽게 초대 행동 발생
- 2016년: 가입 유저 2,500만 → 리퍼럴 마케팅이 Growth의 핵심

**PO 교훈**: **앱 내 가치가 "혼자서는 경험 불가"인 구조**가 가장 강력한 초대 동기. "친구가 없으면 소개를 볼 수 없는" 구조를 설계하면 초대가 자연스럽게 발생한다.

### 4-4. Duolingo Friend Streak (2023 출시)

**Friend Streak 핵심 수치**
- Friend Streak 1개 이상 있는 유저는 **일일 학습 완료율 22% 상승**
- Friend Streak 수가 늘어날수록 완료율 추가 상승
- 전체 유저의 57%가 친구 1명 이상 보유 (사회 기능 기반)

**바이럴 공유 트리거 (3가지)**
1. **연속 마일스톤**: 100일, 365일 달성 → 스크린샷 공유 충동 → 자기 자랑 + 앱 노출
2. **황당한 문장**: 특이한 번역 문장이 밈이 됨 → 유머 콘텐츠로 공유
3. **리더보드 1~3위**: 주간 리그 상위권 → 경쟁심 + 증명 욕구 → 공유

**설계 결정 인사이트**
- Duolingo는 "파워 유저만 공유한다"가 아니라 **"초대 단계 전환율 1% 개선이 파워 유저 최적화보다 더 많은 노출 증가"**임을 깨닫고 초대 단계 UX 반복 개선에 집중

**PO 교훈**
1. 친구 기능은 리텐션 기능이기도 하다 — 연결된 유저는 혼자인 유저보다 22% 더 행동
2. **공유 트리거는 "성취" + "유머" + "경쟁"** — 이 중 하나를 충족하는 순간을 설계해야 함
3. 초대 퍼널의 초기 단계 UX를 반복 개선하는 것이 파워 유저 전환보다 임팩트가 크다

### 4-5. Strava (2009~현재)

**핵심 소셜 메카닉**
- **Kudos (쿠도스)**: 친구 활동에 "좋아요" 개념 → 달성을 인정받는 사회적 보상
- **Segments + Leaderboard**: 특정 구간 최고 기록 경쟁 → 세그먼트가 있는 도로를 달릴 때마다 친구와 경쟁 발생
- **Club**: 위치/스포츠 기반 그룹 → 그룹 내 공유 활동 → 커뮤니티 형성

**성장 수치**
- 최초 10억 건 활동 기록: 8년 소요
- 그 다음 10억 건: 6개월 소요 (소셜 기능 + 커뮤니티 성숙 효과)

**PO 교훈**
- 친구의 활동이 **내 행동의 동기**가 되는 구조 → 소셜 기능이 코어 프로덕트 루프에 내재화
- 리더보드 경쟁은 **자연적인 공유 충동**을 만듦 — "나 오늘 1위야" 스크린샷이 신규 유저 유입

### 4-6. Venmo (2009~현재)

**소셜 피드가 핵심 바이럴 메카닉인 이유**

Venmo는 결제 앱이지만 **소셜 피드**가 성장의 핵심 드라이버였다.

- 거래 내역이 기본 공개 → 이모지와 한마디 설명이 소셜 피드에 노출
- 예시: "for pizza and beer on Friday" → 비유저가 친구의 피드를 보면서 자연스럽게 인식
- "내 친구들이 쓰는 앱이다" 인식 → 설치 동기 유발
- 초기 타겟: 펜실베이니아대 학생 → 졸업 후 alumni 네트워크로 자연 확산

**성장 메카닉 분해**
```
각 거래 → 피드에 공개 (기본값)
→ 비유저 친구가 피드 확인 (링크 타고 앱 페이지 이동)
→ "친구들이 다 쓰네" 인식
→ 설치
→ 새 유저가 거래 → 피드 공개 → 루프 반복
```

**PO 교훈**: "소개 완료" 알림, "OO님이 당신을 주선해줬어요" 같은 외부 가시성 이벤트가 있으면 → 비유저 유입 루프 가능. 단, **프라이버시 기본값은 항상 opt-in**으로 설계해야 함.

### 4-7. Snapchat (2011~현재)

**친구 그래프가 코어 프로덕트인 이유**
- 스냅은 **받는 사람을 지정해서 보내야 함** → 친구가 없으면 앱이 작동하지 않음
- 사라지는 메시지 + 읽음 확인 → 보낸 사람에게 "읽었다" 알림 → 사회적 의무감 생성
- 2012년: 전체 스냅의 30%가 그룹으로 발송

**현재 기술 인프라**
- NebulaGraph 기반 친구 추천 그래프 엔진
- GiGL (Graph Neural Network): 수억 DAU의 1~2촌 추천 계산

**PO 교훈**: **앱 핵심 기능이 친구 없이는 작동하지 않는 구조** → 가장 강력한 초대 동기. 내친소 "소개 기능은 친구가 있어야 활성화"도 같은 원리.

### 4-8. Dropbox (2008~현재) — 리퍼럴 프로그램 교과서

**핵심 수치**

| 항목 | 수치 |
|---|---|
| 성장율 | 15개월 만에 10만 → 400만 유저 (+3,900%) |
| 바이럴 계수(K-factor) | 0.35 (매 10명이 3.5명 추가 유입) |
| 피크 기여율 | 일일 신규 가입의 35%가 리퍼럴에서 발생 |
| 2010년 4월 초대 발송 수 | 280만 건/월 |

**양면 보상 구조**
- 추천자 + 추천받은 사람 모두 500MB 스토리지 지급
- 최대 16GB까지 적립 가능

**핵심 설계 원칙**
1. **앱 내부 가치(스토리지)로 보상** — 현금이 아니라 제품 자체의 핵심 가치
2. **양면 보상** — 추천자만 받는 게 아니라 받는 사람도 혜택 → 죄책감 없이 초대 가능
3. **계단식 적립** — 최대 한도까지 계속 초대 동기 유지

**PO 교훈**: K-factor 0.35도 유료 광고 없이 39배 성장을 만들 수 있다. K > 1이 아니어도 충분히 의미 있다. 내친소에서 **썬구리로 양면 보상**하는 아이디어의 근거.

### 5-1. LinkedIn "Get Introduced" + 상호 연결 표시

**Get Introduced 플로우**
```
1. 2촌 프로필 방문 → "Get Introduced" 버튼 클릭
2. 중간 연결 친구 선택
```

**핵심 수치**
- 상호 연결 있는 경우 콜드 메시지 대비 **응답률 70% 상승**
- 와튼 비즈니스 스쿨 연구: 리퍼럴 고객의 CAC가 비리퍼럴 대비 **$23.12 저렴**
- Warm intro 미팅 전환율: 콜드 아웃리치 대비 **3~5배 높음**

**"How You're Connected" 기능**
- 프로필에 1촌/2촌 공통 지인 즉시 표시
- 공통 지인 이름 표시 → "소개 부탁해도 될 사람이 있다"는 맥락 제공

**PO 교훈**

### 6-1. K-factor (바이럴 계수)

```
K = 사용자당 평균 초대 수 x 초대→가입 전환율

K > 1.0  → 자생적 바이럴 성장
K 0.3~0.9 → 유료 채널 비용 30~90% 절감
```

| 앱 | K-factor | 결과 |
|---|---|---|
| Dropbox | ~0.35 | 15개월 +3,900% (10만 → 400만) |
| Locket Widget | 추정 >1.0 | 2주 200만 가입, 광고비 ≈ 0 |
| Clubhouse | 추정 >1.0 (단기) | 2개월 60만 → 1,000만 |
| BeReal | 추정 0.5~1.0 (초기) | 1년 내 7,000만 피크, 이후 급락 |
| Cerca | 추정 0.5~0.8 | 3개월 85,000 (연락처 기반 자연 확산) |

**데이팅앱에서 K > 1이 어려운 이유**:
매칭 성공 → 두 명 동시 이탈 → 전파자 풀이 지속적으로 축소됨. 주선자를 유지해야 하는 구조적 이유.

## 7. MVP 스펙 결정 근거

> 아래 8개 항목은 코어 루프 개발 전에 확정해야 하는 스펙이다.
> 각 항목마다 "왜 이렇게 해야 하나"의 레퍼런스 근거를 붙였다.
> **Cerca를 핵심 벤치마크로 활용하되, 내친소만의 차별점을 반영한다.**

### (1) 역할 통합: 피드 하나에 합친다

| 앱 | 구조 | 결과 |
|---|---|---|
| Ship (Match Group) | crew 역할 분리 — 주선자는 별도 화면 | 매칭 53%가 crew발이었으나 **주선자 이탈로 2022 종료** |
| Wingman | 같은 앱에서 "나도 매칭 + 친구도 매칭" | 생존 중. Wingman 있으면 매칭 80% 증가 |
| Hinge Matchmaker (2017) | 별도 앱 → 나중에 본앱 통합 | 별도 앱일 때 DAU 부족 → 통합 후에야 작동 |
| **Cerca** | 주선자 역할 없음 (자동화) | 주선자 이탈 문제 자체가 없음 — 그러나 인간적 터치도 없음 |

- Ship 실패 원인 = UX가 아니라 구조. 주선자에게 self-serving 이유가 없었음
- **→ 같은 피드에서 "좋아요" + "친구에게 소개" 두 액션 모두 가능하게**
- Cerca는 이 문제를 "자동화"로 회피했지만, 내친소는 주선자의 능동적 참여가 핵심이므로 **역할 통합이 필수**

### (2) 소개 버튼: 카드에 항상 노출

| 레퍼런스 | 위치 | 효과 |
|---|---|---|
| TikTok | 영상 옆 사이드바 (항상 노출) | 공유율 업계 최고 |
| Hinge "Send Profile" | 점 3개 메뉴 안 | 접근성 낮음 → 실사용 적음 |
| Instagram Reels | 하단 DM 아이콘 | TikTok과 유사, 높은 공유율 |
| **Cerca** | 자동 매칭 (소개 버튼 불필요) | 구조적으로 다름 |

- 메뉴 안에 숨기면 안 됨 (Hinge = 기능 있는데 안 씀)
- **→ 카드 하단 또는 좋아요 버튼 옆에 항상 노출. 라벨: "이 사람 어때?"**

### (3) 추천 메시지: 프리셋 + 편집 가능

| 포맷 | 레퍼런스 | 핵심 수치 |
|---|---|---|
| 이모지만 | Instagram Quick React | 마찰 최소, 정보량 0 |
| 자유 텍스트 | KakaoTalk 소개팅 | Hinge: 63%가 뭘 써야 할지 모름 |
| 프리셋 문장 (편집 가능) | AngelList 소개 템플릿 | 응답률 50% |

- Hinge: 텍스트 프롬프트 → 사진 좋아요 대비 첫 데이트 전환율 47% 높음
- 그러나 63%가 빈칸 앞에서 멈춤 → 기본 문장 제공 필수
- **→ "얘 진짜 좋은 사람이야, [한 줄 추가]" 형태. 이모지는 보조**
- Cerca에는 이 기능이 없음 → **내친소만의 차별화 포인트**

### (4) 동의 구조: 소개 전 opt-in

| 앱 | 방식 | 반응 |
|---|---|---|
| Hinge Matchmaker (2017) | 동의 없이 소셜 그래프 노출 (opt-out) | "creepy" → 별도 앱 실패 직접 원인 |
| LinkedIn 상호 소개 | 양쪽 "소개 받을래요?" 먼저 물음 | 수용됨 |
| **Cerca** | 연락처 전체 동기화 (opt-in이지만 범위가 넓음) | 프라이버시 우려 존재 |
| **Raya** | 추천인 + 심사 (강한 opt-in) | 독점성으로 작동 |

- **→ 주선자가 버튼 탭 → 상대방에게 "OO님이 소개하고 싶대요. 수락?" 알림 → 수락 시 프로필 공개**
- "소개받고 싶어요" 토글 = opt-in의 상위 개념
- Cerca보다 **프라이버시 보호가 강함** = 내친소 차별화

### (5) 보상(썬구리): 행동 단계별 분산

| 레퍼런스 | 보상 시점 | 효과 |
|---|---|---|
| Dropbox | 가입 완료 시, 양쪽 500MB | 15개월 3,900% 성장, CAC 60% 절감 |
| Uber (인도) | 첫 탑승 완료 시, 양쪽 $10-30 | ROI 12배 |
| PayPal | 가입 즉시 $20 → 점진 축소 | 허수 유입 문제 → $20→$5 축소 |
| **Raya** | 추천인 책임 (보상 대신 책임) | 품질 자정 효과 |

- 너무 이른 보상 = 허수, 너무 늦은 보상 = 동기 부족
- **→ 소개 전송 +5 / 수락 +10 / 대화 시작 +20 / 매칭 성사 +50+뱃지**
- 초기 넉넉 → 유저 풀 확보 후 점진 축소 (PayPal 패턴)
- Raya식 **추천인 신뢰도 시스템** 병행 검토

### (6) 카카오 공유 카드: 추천사 + 블러 사진

| 레퍼런스 | 미리보기 깊이 | 효과 |
|---|---|---|
| Locket | 일부만 노출 → 앱 열어야 전체 | 2주 200만 가입 |
| LinkedIn | 상단 3줄 + "더 보려면 로그인" | 검색 유입 → 가입 핵심 퍼널 |
| **S'More** | 블러 사진 → 대화로 언블러 | 호기심 유발 + 외모 편향 감소 |

- 다 보여주면 설치 이유 없음. 안 보여주면 클릭 안 함
- **→ 보여줄 것: 친구 추천사 + 블러 사진 + 나이/지역. 선명한 사진/상세 프로필은 앱에서만**
- S'More의 블러 리빌 + Locket의 궁금증 구조 결합

### (7) 일일 소개 제한: 하루 3회

| 앱 | 제한 | 효과 |
|---|---|---|
| **Cerca** | 하루 4명 | 선택에 진지함 부여. 3개월 85K 유저 |
| **Coffee Meets Bagel** | 정오 6명 | 10년 이상 유지된 일일 습관 모델 |
| Thursday | 주 1일만 | 초기 화제 → 6일간 앱 사망 → **2025년 앱 셧다운** |
| Tinder | 무제한(유료) | 스와이프 피로 → 유저 피로감 심화 |

- 무제한 = 피로 + 품질 하락. 극단 제한(Thursday) = 습관 실패 → 셧다운
- Cerca 4명 + CMB 6명 벤치마크 → 내친소는 3회가 적절 (소개의 진지함 강조)
- **→ 하루 3회. 썬구리로 추가 소개권 구매 가능 (수익화 연결점)**

### (8) 결과 피드백: 진행은 알려주되 거절 상세는 숨김

| 정보 | 공개 여부 | 근거 |
|---|---|---|
| "수락했어요" | O | 주선자 동기 부여 첫 단계 |
| "대화 시작했어요" | O | 썬구리 +20 트리거 |
| "매칭 성사!" | O + 뱃지 | 최고 보상 + 공유 카드 생성 |
| 대화 내용 | X | 프라이버시 |
| 누가 거절했는지 | X | 관계 훼손 방지 |
| 거절 사실 | 모호하게 | "이번엔 연결이 안 됐어요" (누구 탓인지 안 밝힘) |

- Wingman: 매칭 성사 알림 = 주선자 리텐션 핵심
- Hinge: 매칭 후 70%가 메시지 안 보냄 → "대화 시작" 알림이 가치 있음
- Cerca: 익명 좋아요로 거절 자체를 인지 불가하게 설계 → 내친소도 거절의 부담을 최소화
