# 기능 1: 피드 즉시 공유 ("이 사람 어때?")

> 피드 카드 → "친구에게 소개" 버튼 → 친구 선택 → 한 줄 메모(선택) → 전송. **3초 이내 완료**
> 출처: TikTok 영상 공유, Instagram DM 공유, Hinge "Send Profile"

## 1-2. DB 스키마

### 테이블: `referrals` (소개 기록)

```sql
CREATE TABLE referrals (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    sender_id       UUID NOT NULL REFERENCES users(id),        -- 소개하는 사람 (주선자)
    receiver_id     UUID NOT NULL REFERENCES users(id),        -- 소개받는 친구
    target_user_id  UUID NOT NULL REFERENCES users(id),        -- 소개 대상 프로필
    message_type    VARCHAR(20) NOT NULL DEFAULT 'none',       -- 'preset_1'~'preset_4', 'custom', 'none'
    custom_message  VARCHAR(50),                                -- 자유입력 메시지 (message_type='custom'일 때)
    status          VARCHAR(20) NOT NULL DEFAULT 'sent',       -- 'sent', 'viewed', 'accepted', 'declined', 'expired'
    sungguri_given  BOOLEAN NOT NULL DEFAULT FALSE,            -- 썬구리 지급 여부
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    viewed_at       TIMESTAMPTZ,
    responded_at    TIMESTAMPTZ,

    -- 같은 조합 중복 소개 방지 (24시간 이내)
    CONSTRAINT uq_referral_daily UNIQUE (sender_id, receiver_id, target_user_id)
);

-- 인덱스
CREATE INDEX idx_referrals_sender ON referrals(sender_id, created_at DESC);
CREATE INDEX idx_referrals_receiver ON referrals(receiver_id, status, created_at DESC);
CREATE INDEX idx_referrals_target ON referrals(target_user_id);
```

### 테이블: `preset_messages` (프리셋 메시지 관리)

```sql
CREATE TABLE preset_messages (
    id          SERIAL PRIMARY KEY,
    text        VARCHAR(50) NOT NULL,
    sort_order  INT NOT NULL DEFAULT 0,
    is_active   BOOLEAN NOT NULL DEFAULT TRUE,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- 초기 데이터
INSERT INTO preset_messages (text, sort_order) VALUES
    ('너 스타일이야 ㅋㅋ', 1),
    ('취향 저격 아니야?', 2),
    ('진지할 것 같아서', 3),
    ('한번 봐봐!', 4);
```

### 테이블: `referral_stats` (주선 통계 캐시)

```sql
CREATE TABLE referral_stats (
    user_id             UUID PRIMARY KEY REFERENCES users(id),
    total_sent          INT NOT NULL DEFAULT 0,
    total_accepted      INT NOT NULL DEFAULT 0,
    total_sungguri      INT NOT NULL DEFAULT 0,
    last_sent_at        TIMESTAMPTZ,
    updated_at          TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### ER 다이어그램 (텍스트)

```
users ──1:N──▶ referrals (sender_id)
users ──1:N──▶ referrals (receiver_id)
users ──1:N──▶ referrals (target_user_id)
users ──1:1──▶ referral_stats
```

## 1-3. API 엔드포인트

### POST `/api/v1/referrals` — 소개 전송

```
Method: POST
Path:   /api/v1/referrals
Auth:   Bearer Token (sender)
```

**Request Body:**

```json
{
    "receiver_ids": ["uuid-1", "uuid-2"],
    "target_user_id": "uuid-target",
    "message_type": "preset_1",
    "custom_message": null
}
```

**Response (201 Created):**

```json
{
    "success": true,
    "data": {
        "referral_count": 2,
        "sungguri_earned": 5,
        "referral_ids": ["ref-uuid-1", "ref-uuid-2"]
    }
}
```

**에러 응답:**

| HTTP Code | 상황 | 응답 |
|-----------|------|------|
| 400 | receiver_ids 빈 배열 | `{"error": "EMPTY_RECEIVERS"}` |
| 400 | receiver_ids > 5명 | `{"error": "TOO_MANY_RECEIVERS"}` |
| 400 | 자기 자신에게 소개 | `{"error": "SELF_REFERRAL"}` |
| 403 | target_user 소개 토글 OFF | `{"error": "USER_NOT_OPEN"}` |
| 409 | 24시간 내 동일 조합 중복 | `{"error": "DUPLICATE_REFERRAL"}` |
| 429 | 일일 전송 한도 초과 (50건) | `{"error": "DAILY_LIMIT"}` |

## 1-4. 엣지 케이스

| # | 상황 | 처리 방법 |
|---|------|----------|
| E1 | **소개 대상이 "소개받고 싶어요" 토글 OFF** | "친구에게 소개" 버튼 비활성화. 탭 시 "이 사람은 지금 소개를 받고 있지 않아요" 토스트 |
| E2 | **같은 사람을 같은 친구에게 24시간 내 재소개** | 409 에러 + "어제 이미 민준에게 소개했어요" 안내 |
| E3 | **소개 후 대상 유저가 탈퇴** | 받은 소개 목록에서 "탈퇴한 사용자" 표시. 프로필 열람 불가 |
| E4 | **소개 후 sender-receiver 친구 관계 해제** | 이미 전송된 소개는 유지. 새 소개만 차단 |
| E5 | **receiver가 앱 미설치 상태** | 신뢰+확산 카카오톡 소개 카드(#5)로 연결. MVP에서는 앱 내 친구만 대상 |
| E6 | **일일 전송 한도 50건 초과** | 429 에러 + "오늘 소개를 많이 했어요! 내일 다시 해주세요" |
| E7 | **네트워크 오류로 전송 실패** | 로컬 큐에 저장 → 재연결 시 자동 재전송. 실패 토스트 + 재시도 버튼 |
| E8 | **복수 친구 선택 시 일부만 전송 성공** | 성공건은 즉시 반영. 실패건은 "1명에게 전송 실패" 안내 + 재시도 |
| E9 | **차단한 유저의 프로필이 소개로 도착** | receiver가 target을 차단한 경우 해당 소개 자동 필터링 (목록에 미노출) |
| E10 | **30일간 미응답 소개** | `expired` 상태로 자동 전환. 배치 처리 (매일 02:00 KST) |

## 1-5. 썬구리 보상 연동

| 이벤트 | 수혜자 | 보상 | 지급 시점 | 중복 지급 |
|--------|--------|------|----------|----------|
| 소개 전송 | 주선자 (sender) | +5 🌟 | 전송 즉시 | 동일 조합 24시간 내 1회 |
| 프리셋/자유입력 메시지 첨부 | 주선자 (sender) | +3 🌟 (보너스) | 전송 즉시 | 메시지 없이 보내면 미지급 |
| "볼게!" 수락 | 소개받는 사람 (receiver) | +10 🌟 | 수락 즉시 | 소개 건당 1회 |
| "볼게!" 수락 알림 | 주선자 (sender) | 알림만 (보상은 #15 피드백 루프에서) | 수락 시 | - |

### 어뷰징 방지

```
일일 소개 전송 한도: 50건
일일 썬구리 획득 상한: 250 🌟 (소개 전송 기준)
동일 조합 중복 방지: 24시간 쿨다운
자기 자신 소개 차단: API 레벨에서 거부
```

### 정보 공개 순서 (3단계)

| 단계 | 보이는 것 | 안 보이는 것 | 의도 |
|------|----------|------------|------|
| Step 1 | 추천사 + 주선자 이름 | 프로필 정보, 사진 | "누가 소개했는지"에 집중 |
| Step 2 | + 기본 정보 (나이, 직업, 지역, 자기소개) | 사진 | "어떤 사람인지"에 집중 |
| Step 3 | + 사진 전체 | 없음 | "외모 확인"은 마지막 |

### "볼게!" / "나중에" 버튼

- **어느 단계에서든 결정 가능**: Step 1에서 추천사만 보고도 "볼게!" 수락 가능
- "볼게!": 양방향 관심 표시 → 매칭 프로세스 진행
- "나중에": 24시간 후 다시 알림 1회 → 이후 만료

## 4-2. DB 스키마

### 테이블: `recommendations` (추천사)

```sql
CREATE TABLE recommendations (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    writer_id       UUID NOT NULL REFERENCES users(id),   -- 추천사 작성자 (= 주선자)
    target_id       UUID NOT NULL REFERENCES users(id),   -- 추천 대상
    content         VARCHAR(200) NOT NULL,                 -- 추천사 본문 (200자 제한)
    relationship    VARCHAR(30),                           -- 관계 태그: '고등학교 동창', '대학 친구', '직장 동료' 등
    is_active       BOOLEAN NOT NULL DEFAULT TRUE,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    -- 한 사람이 같은 친구에게 추천사 1개만
    CONSTRAINT uq_recommendation UNIQUE (writer_id, target_id)
);

CREATE INDEX idx_recommendations_target ON recommendations(target_id, is_active);
```

### 테이블: `referral_views` (소개 카드 조회 단계 추적)

```sql
CREATE TABLE referral_views (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    referral_id     UUID NOT NULL REFERENCES referrals(id),
    viewer_id       UUID NOT NULL REFERENCES users(id),     -- 소개받는 사람
    max_step        INT NOT NULL DEFAULT 1,                 -- 1: 추천사만, 2: 기본정보, 3: 사진
    step1_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),     -- 추천사 조회 시각
    step2_at        TIMESTAMPTZ,                            -- 기본정보 조회 시각
    step3_at        TIMESTAMPTZ,                            -- 사진 조회 시각

    CONSTRAINT uq_referral_view UNIQUE (referral_id, viewer_id)
);

CREATE INDEX idx_referral_views_referral ON referral_views(referral_id);
```


## 4-3. API 엔드포인트

### POST `/api/v1/recommendations` — 추천사 작성

```
Method: POST
Path:   /api/v1/recommendations
Auth:   Bearer Token (writer)
```

**Request Body:**

```json
{
    "target_id": "uuid-target",
    "content": "걔 진짜 성격 좋아. 같이 있으면 시간이 금방 가",
    "relationship": "대학 친구"
}
```

**Response (201 Created):**

```json
{
    "success": true,
    "data": {
        "id": "rec-uuid",
        "sungguri_earned": 8
    }
}
```

**에러 응답:**

| HTTP Code | 상황 | 응답 |
|-----------|------|------|
| 400 | content 비어있음 또는 200자 초과 | `{"error": "INVALID_CONTENT"}` |
| 403 | target과 친구 관계가 아님 | `{"error": "NOT_FRIENDS"}` |
| 409 | 이미 해당 친구에 대한 추천사 존재 | `{"error": "ALREADY_EXISTS"}` |

## 4-4. 엣지 케이스

| # | 상황 | 처리 방법 |
|---|------|----------|
| E1 | **주선자가 추천사를 안 쓴 상태에서 소개** | Step 1에 추천사 대신 "민준이가 소개해줬어요" + 소개 메시지만 표시. 추천사 영역은 비움 |
| E2 | **추천사가 여러 개 있는 경우** | 소개한 주선자의 추천사를 우선 표시. 추가 추천사는 Step 2에서 "다른 친구들의 추천" 섹션에 노출 |
| E3 | **추천사 내 비속어/부적절 내용** | 작성 시 금칙어 필터 적용. 신고 접수 시 즉시 비활성화 (`is_active = false`) + 작성자 경고 |
| E4 | **추천사 작성자가 탈퇴** | 추천사 유지하되 작성자를 "탈퇴한 친구"로 표시. 삭제 요청 시 `is_active = false` |
| E5 | **Step 1에서 바로 "볼게!" 누른 경우** | 유효한 액션. Step 2, 3을 건너뛰고 바로 수락 처리. `referral_views.max_step = 1`로 기록 |
| E6 | **사진이 없는 유저** | Step 3 건너뛰고 Step 2까지만 표시. "사진 보기" 버튼 대신 "프로필이 전부예요" 안내 |
| E7 | **추천사 200자 초과 시도** | 프론트에서 실시간 글자수 카운터. 200자 도달 시 입력 차단. API에서도 재검증 |
| E8 | **"나중에" 후 재알림** | 24시간 후 1회 리마인드 알림. 이후 미응답 시 30일 뒤 만료 (기능 1의 E10과 동일) |
| E9 | **Step 중간에 앱 종료** | `referral_views.max_step` 기록으로 복원. 다음 진입 시 마지막 본 단계부터 재개 |
| E10 | **동일 target에 대해 여러 명이 소개** | 각 소개는 독립 카드. 단, 목록에 "3명의 친구가 이 사람을 소개했어요" 뱃지 표시 |

## 4-5. 썬구리 보상 연동

| 이벤트 | 수혜자 | 보상 | 지급 시점 | 비고 |
|--------|--------|------|----------|------|
| 추천사 작성 | 작성자 | +8 🌟 | 작성 완료 즉시 | 대상 1인당 1회 |
| 소개받은 카드에서 "볼게!" 수락 | 소개받는 사람 (receiver) | +10 🌟 | 수락 즉시 | 기능 1과 동일 |
| 수락 알림 → 주선자 | 주선자 (sender) | (기능 #15에서 지급) | - | 이 기능에서는 알림만 |
| Step 3(사진)까지 조회 후 수락 | 소개받는 사람 | +2 🌟 (보너스) | 수락 시 | "꼼꼼히 봤네요" 보너스 |

### 추천사 작성 인센티브 구조

```
추천사 미작성 → 소개 시 카드가 밋밋해짐 → 수락률 하락
추천사 작성   → +8 썬구리 + 소개 카드가 풍성해짐 → 수락률 상승
                → 간접적으로 매칭 성사 확률 ↑ → 피드백 루프 보상(+20, +50) 기대
```

| 행동 조합 | 총 획득 썬구리 |
|----------|---------------|
| 추천사 작성만 | +8 🌟 |
| 추천사 작성 + 소개 전송 (메시지 포함) | +8 + 5 + 3 = +16 🌟 |
| 위 + 상대 수락 + 대화 시작 + 매칭 성사 | +16 + 20 + 50 = +86 🌟 |

## 부록: 두 기능 간 의존 관계

```
기능 1 (피드 즉시 공유) ──의존──▶ 기능 4 (추천사 먼저 보기)

- 기능 1의 소개 전송(referrals) → 기능 4의 추천사 카드로 수신

- 두 기능은 동시 개발 가능하지만 통합 테스트 필수
```

|------|--------|-----------|

| 2주차 | 기능 4 API (단계별 카드 응답), 썬구리 통합 | 기능 4 전체 (추천사 카드 + 순차 공개) |
